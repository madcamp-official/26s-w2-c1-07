## 1. 목적

현재의 단순한 `종료 시점 높이 점수 + 고정 순위 보너스` 방식을 확장하여 다음 플레이를 함께 보상한다.

- 라운드 중 실제로 얼마나 높이 올라갔는가
- 라운드 종료 시 높은 위치를 얼마나 잘 유지했는가
- 얼마나 빠르게 정상 및 각 청크에 도달했는가
- 최종적으로 몇 위를 기록했는가
- 반복 탈락 없이 안정적으로 등반했는가
- 자신이 설치한 투명 구조물로 상대 플레이에 영향을 주었는가

## 2. 유지할 시스템 제약

이번 개편에서는 실시간 점수 계산 및 실시간 점수 동기화를 구현하지 않는다.

- 라운드 진행 중 호스트가 점수 계산에 필요한 사건과 통계만 기록한다.
- `Play → Highlight` 전환 시 호스트가 라운드 점수를 최종 계산한다.
- 호스트가 확정한 결과를 다른 클라이언트에 전파하는 현재 권위 구조를 유지한다.
- 현재는 호스트 컴퓨터에서 서버가 동작하는 구조를 유지한다.
- 향후 전용 서버 및 실시간 점수 계산으로 이전할 수 있도록 계산 로직과 데이터 수집 로직은 분리한다.
- 최종 점수 공식은 가능한 한 단일 계산 지점에 유지한다.

## 3. 라운드 점수 구성

```text
라운드 점수 =
    진행도 점수
    + 정상 도달 시간 점수
    + 청크 선착순 보너스
    + 참가자 수 비례 순위 점수
    + 안정성 보너스
    + 투명 구조물 영향 점수
    - 반복 탈락 감점
```

최종 라운드 점수가 음수가 될 수 있는지는 아직 미정이다. 기본 권장안은 최종 결과를 `0점 이상`으로 제한하는 것이다.

## 4. 점수 항목별 규칙

### 4.1 진행도 점수

라운드 중 최고 도달 높이와 라운드 종료 시점 높이를 모두 반영한다.

```text
진행도 점수 =
    최고 높이 점수 × 70%
    + 최종 높이 점수 × 30%
```

각 높이 점수는 높이를 맵 정상 높이로 정규화하여 계산한다.

```text
정규화 높이 = Clamp01(대상 높이 / 맵 정상 높이)
높이 점수 = 높이 점수 만점 × 높이 점수 곡선(정규화 높이)
```

판정 원칙:

- `최고 높이`는 라운드 중 플레이어가 기록한 가장 높은 발끝 높이다.
- `최종 높이`는 라운드 종료 시점에 채점 대상으로 결정된 발끝 높이다.
- 정상 도달자는 최고 높이를 정상 높이로 고정한다.
- 최고 높이는 탈락 및 부활 후에도 유지한다.
- 동일 라운드에서 최고 높이가 낮아지는 일은 없다.
- 기존 `AnimationCurve` 기반 높이 가치 조정 기능은 유지하는 것을 권장한다.

### 4.2 정상 도달 시간 점수

정상에 도달한 플레이어만 받으며, 빨리 도달할수록 높은 점수를 받는다.

```text
시간 점수 =
    정상 도달 여부
    × 최대 시간 점수
    × 남은 시간 / 전체 라운드 시간
```

동일한 의미의 표현:

```text
시간 점수 =
    정상 도달 시 최대 시간 점수
    × Clamp01(1 - 정상 도달 시간 / 전체 라운드 시간)
```

판정 원칙:

- 정상 미도달자는 0점이다.
- 정상 도달 시각은 최초 도달 시각만 인정한다.
- 정상 도달 후 떨어지거나 탈락해도 최초 도달 기록은 유지한다.
- 현재 수집 중인 `ReachedTop`, `TopTime` 데이터를 활용할 수 있다.
- `finishGrace`로 라운드 시간이 단축되더라도 분모는 최초 설정된 전체 라운드 시간을 사용하는 것을 기본안으로 한다.

예시: 전체 라운드 180초, 최대 시간 점수 300점

| 정상 도달 시점 | 남은 시간 | 시간 점수 |
|---:|---:|---:|
| 30초 | 150초 | 250점 |
| 60초 | 120초 | 200점 |
| 120초 | 60초 | 100점 |
| 미도달 | - | 0점 |

### 4.3 청크 선착순 보너스

맵을 높이 또는 진행 구간 기준의 청크로 나누고, 각 청크에 도달한 순서에 따라 보너스를 지급한다.

```text
청크 최초 도달: +40점
청크 두 번째 도달: +20점
청크 세 번째 도달: +10점
청크 네 번째 이후 도달: 0점
```

판정 원칙:

- 선착순은 각 청크마다 독립적으로 판정한다.
- 플레이어별로 같은 청크의 최초 도달만 인정한다.
- 탈락 후 같은 청크에 재진입해도 보너스를 다시 주지 않는다.
- 청크를 건너뛰어 상위 청크에 진입한 경우, 건너뛴 하위 청크까지 도달로 볼지는 별도 결정이 필요하다.
- 동시 도달 판정은 호스트가 기록한 서버 시각을 우선 사용하고, 완전히 같으면 `ClientId` 등 결정론적 기준을 사용한다.
- 청크 경계는 랜덤 맵 생성 결과와 무관하게 모든 피어에서 같은 의미를 가져야 한다.

현재 프로젝트에는 명시적인 청크/체크포인트 점수 도메인이 없으므로 새 데이터 정의와 도달 기록이 필요하다.

### 4.4 참가자 수 비례 순위 점수

기존의 고정 `300/200/100` 보너스 대신 참가자 수에 따라 순위 점수를 선형 배분한다.

```text
순위 점수 =
    최대 순위 점수
    × (참가자 수 - 순위)
    / (참가자 수 - 1)
```

- 순위는 1부터 시작한다.
- 1위는 최대 순위 점수를 받는다.
- 최하위는 0점을 받는다.
- 중간 순위는 참가자 수에 맞춰 선형으로 배분한다.
- 참가자가 1명이면 0으로 나누지 않도록 별도 처리한다.

예시: 5명, 최대 순위 점수 300점

| 순위 | 순위 점수 |
|---:|---:|
| 1위 | 300점 |
| 2위 | 225점 |
| 3위 | 150점 |
| 4위 | 75점 |
| 5위 | 0점 |

동점 처리 방식은 아직 미정이다. 다음 중 하나를 명시적으로 선택해야 한다.

1. 기존처럼 정상 도달 시각 및 결정론적 식별자로 순위를 나눈다.
2. 공동 순위를 인정하고 해당 순위 구간 점수의 평균을 지급한다.

### 4.5 안정성 보너스

충분히 등반하면서 한 번도 탈락하지 않은 플레이어에게 보너스를 지급한다.

```text
최고 높이 60% 이상
+ 탈락 0회
→ 안정성 보너스 지급
```

판정 원칙:

- `최고 높이 / 맵 정상 높이 >= 0.6`이어야 한다.
- 라운드 탈락 횟수가 정확히 0회여야 한다.
- 시작 지점에서 움직이지 않고 버티는 플레이는 보상하지 않는다.
- 안정성 보너스 금액은 설정값으로 둔다.

### 4.6 반복 탈락 감점

첫 탈락은 감점하지 않고 반복 탈락부터 점수를 차감한다.

```text
첫 탈락: 0점
두 번째 탈락: -10점
세 번째부터: 탈락 1회당 -20점
```

누적 감점 예시:

| 총 탈락 횟수 | 총 감점 |
|---:|---:|
| 0회 | 0점 |
| 1회 | 0점 |
| 2회 | -10점 |
| 3회 | -30점 |
| 4회 | -50점 |
| 5회 | -70점 |

공식 예시:

```text
탈락 횟수 <= 1: 0
탈락 횟수 == 2: 10점 감점
탈락 횟수 >= 3: 10 + (탈락 횟수 - 2) × 20점 감점
```

권장 안전장치:

- 라운드당 최대 감점 한도를 설정한다.
- 최종 라운드 점수를 0점 이상으로 제한할지 결정한다.
- 연결 끊김, 강제 이동, 서버 오류 등 플레이어 책임이 아닌 탈락은 제외할 수 있어야 한다.

### 4.7 투명 구조물 영향 점수

자신이 설치한 투명 구조물이 다른 플레이어의 탈락에 영향을 주면 설치자에게 점수를 지급한다.

```text
상대가 내 투명 구조물에 접촉
+ 접촉 후 일정 시간 안에 상대가 탈락
→ 설치자 +40점
```

판정 원칙:

- 피해자와 구조물 설치자가 다를 때만 인정한다.
- 자기 구조물에 자신이 당한 경우 점수를 지급하지 않는다.
- 접촉과 탈락 사이에 유효 시간 제한을 둔다.
- 하나의 접촉 기록은 하나의 탈락에만 귀속한다.
- 가장 최근의 유효한 투명 구조물 접촉을 탈락 원인 후보로 사용한다.
- 동일 피해자를 반복해서 이용하는 점수 파밍 방지 규칙이 필요하다.
- 구조물 영향 점수에는 라운드당 획득 상한을 두는 것을 권장한다.
- 기존 `MatchStatsTracker`의 투명 구조물 접촉 및 낚시 통계를 재사용할 수 있는지 우선 검토한다.

## 5. 필요한 라운드 성적 데이터

플레이어별로 최소한 다음 데이터를 라운드 동안 호스트가 기록해야 한다.

```text
ClientId
최고 도달 높이
라운드 종료 채점 높이
정상 도달 여부
최초 정상 도달 시각
청크별 최초 도달 시각 또는 도달 순번
최종 순위
총 탈락 횟수
유효한 투명 구조물 영향 횟수
```

권장 분리:

- `수집 데이터`: 라운드 중 발생한 사실
- `점수 설정`: 만점, 가중치, 감점, 유효 시간, 상한
- `점수 계산`: 수집 데이터와 설정을 받아 결과를 반환하는 순수 함수
- `네트워크 결과`: 최종 확정된 라운드 점수 및 표시 데이터

## 6. 점수 설정값 후보

다음 값은 코드 상수가 아니라 인스펙터 또는 별도 설정 객체에서 조정 가능하게 만드는 것을 권장한다.

```text
진행도 점수 만점
최고 높이 가중치 = 0.7
최종 높이 가중치 = 0.3
높이 점수 곡선
정상 도달 시간 점수 만점
시간 점수 곡선
청크별 선착순 보너스 배열 = [40, 20, 10]
순위 점수 만점
안정성 최소 높이 비율 = 0.6
안정성 보너스
두 번째 탈락 감점 = 10
세 번째 이후 탈락당 감점 = 20
라운드 최대 탈락 감점
투명 구조물 영향 점수 = 40
투명 구조물 영향 인정 시간
투명 구조물 영향 점수 상한
라운드 최종 점수 최솟값
```

## 7. 아직 결정되지 않은 사항

구현 전에 다음 내용을 사용자와 협의하거나 명시적으로 가정해야 한다.

1. 각 점수 항목의 최대 점수와 전체 비중
2. 청크의 개수, 경계 및 식별 방식
3. 상위 청크 진입 시 건너뛴 하위 청크도 도달 처리할지 여부
4. 청크 동시 도달 처리 방식
5. 최종 순위의 동점 처리 방식
6. 안정성 보너스 금액
7. 탈락 감점 상한
8. 최종 라운드 점수의 최소값을 0으로 제한할지 여부
9. 투명 구조물 접촉 후 탈락 인정 시간
10. 동일 피해자 반복 낚시 제한 방식
11. 투명 구조물 영향 점수의 라운드당 상한
12. `finishGrace` 적용 후 시간 점수 계산의 기준 시간
13. 라운드 결과 UI에서 총점만 보여줄지 항목별 내역도 보여줄지 여부

## 8. Claude Code 작업 요청 시 사용할 프롬프트

아래 요구사항을 기준으로 현재 Unity 프로젝트의 점수 시스템 개편을 설계해줘.

1. 먼저 현재 점수 계산, 라운드 성적 수집, 네트워크 복제, 점수판 및 하이라이트 표시 흐름을 코드에서 다시 확인해라.
2. 이번 단계에서는 실시간 점수 계산이나 실시간 점수 동기화를 구현하지 마라.
3. 호스트가 라운드 중 필요한 통계만 수집하고 `Play → Highlight` 전환 시 최종 점수를 계산하는 현재 구조를 유지해라.
4. 이 문서의 점수 항목을 서로 독립적인 계산 항목으로 구성하고, 설정값과 계산식을 한 곳에서 관리할 수 있게 설계해라.
5. 향후 전용 서버 및 실시간 계산으로 이전하기 쉽도록 이벤트 수집, 점수 계산, 네트워크 결과 표현을 분리해라.
6. 기존 네트워크 직렬화 구조를 변경해야 한다면 호환성 영향과 마이그레이션 방안을 먼저 설명해라.
7. 아직 결정되지 않은 사항은 임의로 구현하지 말고, 권장 기본값과 대안을 제시한 뒤 확인을 요청해라.
