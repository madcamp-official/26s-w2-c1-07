# 클라이밍 전환 명세서 (Only Up 모티브)

> 게임 컨셉 전면 전환(팀 결정, 2026-07-11). 기존 "주제 룰렛 + 수평 코스 레이스" 구조를 폐기하고,
> **플레이어들이 구조물을 설치하며 수직 타워를 오르는 클라이밍 게임**으로 바꾼다.
> Only Up! 에서 모티브만 가져오며 진행 방식은 다르다: 매 라운드 시작 전 준비 페이즈에서
> 구조물을 설치하고, **설치한 구조물이 3라운드 내내 누적**되는 것이 핵심 루프다.
> 이 문서가 신규 컨셉의 단일 기준 명세다. 기존 기획서(docs/기획서.md)의 룰렛·주제·모드 부분은 무효.

## 1. 용어 정의

| 용어 | 정의 |
|---|---|
| 매치 | 준비 → 라운드1 → 준비 → 라운드2 → 준비 → 라운드3 → 결과 로 이루어진 한 판 |
| 라운드 | 플레이(등반) + 하이라이트로 이루어진 단위. 매 라운드 앞에 준비 페이즈가 붙는다 |
| 구조물 | 플레이어가 설치할 수 있는 모든 Object. 보이는 구조물 / 보이지 않는 구조물 두 종류 |
| 보이는 구조물 | 생성된 순간부터 어떤 페이즈에서든 모든 플레이어에게 보이는 구조물 |
| 보이지 않는 구조물 | 상호작용이 없을 때: 준비 페이즈에는 설치자에게만 보이고, 그 외 페이즈에는 아무에게도 안 보임. 플레이어와 상호작용(충돌 등)이 생기면 모든 플레이어에게 일시적으로 보임 |
| 시작 섬 | (2026-07-12 개편) 맨 아래(윗면 y=0) 직사각형 섬. 기존 "지면"을 대체. 스폰 위치이며, 섬 영역(여유 포함)에는 구조물 설치 금지. 세로(Z) 길이는 레인 수에 비례. 섬 밖은 낭떠러지 |
| 레인 | (2026-07-12 개편) 시작 섬 동쪽(+X)에서 시작해 올라가는 하나의 대각선 길. 발판을 체인처럼 이어붙여 만든다. 맵 = 레인 여러 개를 Z 방향으로 나란히 배치 |
| 발판 | 레인을 구성하는 단위 지형(PlatformSegment 프리팹, 시드 기반 로컬 생성). 입구/출구 앵커로 연결된다 |
| 발판 영역 | 발판 AABB + 여유 폭(regionPadding). 하이라이트 킬 귀속(추후)과 시작 섬 설치 금지 판정의 공통 단위(`ClimbMapGenerator.Regions`, 시작 섬 = Id 0) |
| 설치 허용 범위 | 맵 footprint(섬 + 전체 레인) + placementPadding. 구조물은 이 안(시작 섬 영역 제외)에만 설치 가능(`IsPlacementAllowed`) |
| 이동 범위 | 맵 footprint + movementPadding(`MovementBounds`). 경계벽 위치이자 PREP 비행 범위. 생성 결과에서 파생되므로 레인 수/길이를 바꾸면 자동 추종 |

## 2. 매치 흐름 (FSM)

```
LOBBY → [PREP → PLAY → HIGHLIGHT → INTERMISSION] x3 → RESULT → LOBBY
(마지막 라운드는 INTERMISSION 대신 RESULT 로 진행)
```

- 기존과의 차이: **ROULETTE 페이즈 삭제**(주제 랜덤 생성 없음), **PREP 이 라운드 루프 안으로** 이동(매 라운드 전에 1회씩, 총 3회).
- INTERMISSION = 하이라이트 연출 후 다음 라운드 PREP 로 곧장 넘어가면 어색해 두는 라운드 간 완충 대기(`intermissionDuration`, 기본 5초). 구현 중 추가된 페이즈.
- HIGHLIGHT 는 각 라운드마다 유지(라운드 결과 카드).
- 구조물 정리 시점: **RESULT → LOBBY 전환 시에만 전체 despawn.** 라운드 사이(HIGHLIGHT → PREP)에는 절대 지우지 않는다(누적이 핵심).
- 맵 랜덤 생성 시점: LOBBY → 첫 PREP 진입 시 시드 확정(매 매치 새 맵).
- **LOBBY = 대기방(2026-07-12 추가)**: 씬에 `LobbyManager`(MatchManager 오브젝트의 컴포넌트)가
  있으면 LOBBY 는 타이머로 진행하지 않고 **호스트의 게임 시작(LobbyManager.StartGameServerRpc →
  MatchManager.StartMatchFromLobby)** 으로만 PREP 진입(서버가 최소 인원·전원 준비 재검증).
  RESULT 종료 후 연결을 유지한 채 LOBBY(대기방)로 복귀하며 준비 상태만 초기화(목록/닉네임 유지).
  `_requireManualStart` 를 끄거나 LobbyManager 가 없으면 기존처럼 `lobbyDuration` 후 자동 시작
  (솔로 테스트용). 상세는 12절.

## 3. 맵 랜덤 생성 (2026-07-12 전면 개편: 슬라이스 살포 → 레인 기반)

"발판을 많이 뿌리다 보니 어디로든 갈 수 있는" 구조를 폐기하고, **길이 명확하게 보이는
Only Up 스타일 대각선 레인**으로 전환. 슬라이스/맵 볼륨(6x6x50)/지면(24x24) 개념 삭제.

- **시작 섬**: 직사각형 섬 1개(윗면 y=0, `islandWidth` x 두께 `islandThickness`). 세로(Z) =
  `(레인 수 - 1) x laneSpacing + islandPadding x 2` — 레인 수에 비례해 자동으로 길어진다.
  플레이어는 섬 위에 Z 축 일렬 스폰. **섬 영역(여유 포함)에는 구조물 설치 금지**(스폰 봉쇄 방지).
  섬 밖은 낭떠러지 — 추락은 낙하 탈락 규칙(4절)이 처리.
- **레인 생성**: 레인마다 파생 시드로, 섬 동쪽 가장자리에서 시작해 발판 `platformsPerLane`개를
  체인으로 연결. 커서(직전 발판 출구 앵커) + 간격(전진 g, 상승 g x `risePerMeter`) = 다음 발판
  입구 앵커. Z 지터(`lateralJitter`, `laneHalfWidth` 안 클램프)로 단조로움만 깬다.
- **간격 2종 + 예산 정규화**: 각 간격은 `blockedGapChance` 확률로 단절(`blockedGapRange`,
  구조물 필수) / 아니면 점프 가능(`jumpableGapRange`). 샘플링 후 **합계를 `(발판 수 - 1) x avgGap`
  으로 정규화** → 개별 간격은 제각각, 레인 총 길이(=정상 높이)는 시드와 무관하게 일정(공정성).
  섬 → 첫 발판은 `startGap` 고정(항상 점프 가능, 정규화 제외).
- **발판 프리팹 풀**: `PlatformSegment`(입구/출구 앵커 + 가중치) 프리팹 배열에서 가중치 랜덤 선택.
  플레이어 설치 구조물(Structure_*)과 **같은 시각 에셋 집합을 공유**하되 래퍼가 다르다:
  발판 = PlatformSegment(NetworkObject 없음, ~3x5), 설치물 = Structure + NetworkObject(~1x2).
  회전(`RotatingPlatform`)/이동(`MovingPlatform`) 발판은 서버 시간의 순수 함수로 결정론 동기화
  (위상은 시드에서). 풀이 비면 프리미티브 평판 폴백(프리팹 연결 전에도 동작).
- **발판 영역**: 발판마다 `PlatformRegion{Id, 레인, 순번, AABB, 여유 AABB}` 기록(시작 섬 = Id 0).
  시드 기반 생성 순서라 전 피어에서 Id 동일 → 추후 하이라이트가 "탈락 위치 → 영역" 귀속만 하면 됨
  (탈락 위치는 서버 통계 `MatchStatsTracker.DeathEvents`에 이미 기록 중).
- **파생 범위**: 설치 허용 범위/이동 범위(경계벽)/정상 높이(`MapHeight` = 가장 낮은 레인 정상)를
  전부 생성 결과 footprint 에서 계산 — 레인 수/길이 변경 시 자동 추종.
- **결정론적 생성:** 호스트가 매치마다 시드 1개를 `NetworkVariable<int> MapSeed` 로 복제하고,
  모든 피어가 같은 시드로 동일한 맵을 로컬 생성(NetworkObject 스폰 없음, late-join 자동 대응).
  생성 지오메트리는 전부 `Ground` 레이어(카메라 오클루전 호환).

## 4. 플레이어 (이동·점프·낙하 탈락) (2026-07-12 개편: 체력 시스템 폐지)

- 이동/시점: 기존 그대로(3인칭 마우스룩 조준 시점, WASD, 커서 잠금).
- **점프력: 스페이스바 점프의 Δy 최댓값 = 1** (`jumpHeight`=1). 점프 가능 간격/기울기 튜닝의 기준.
- **체력 시스템 폐지.** 시작 섬 밖이 전부 낭떠러지가 되면서 낙하 자체가 탈락 조건이 된다.
- **낙하 탈락 2규칙** (둘 다 MatchManager 시리얼라이즈드, 서버 판정):
  - **규칙 ① 공중 낙하**: 낙하 거리(최고점 y - 현재 y)가 `_lethalAirFall`(15) 이상이면 착지 여부와
    무관하게 즉시 탈락. 허공 추락(섬/발판 밖)도 이 규칙이 잡는다.
    판정 주체 = **서버**: PLAY 중 위치 샘플링으로 직접 추적(수직 속도 -3m/s 를 넘지 않는 상태가
    0.3초 유지되면 지지된 것으로 보고 기준점 리셋 → 접지 정보 없이도 자유낙하만 정확히 누적).
  - **규칙 ② 착지 낙하**: 낙하 거리가 `_lethalLandFall`(7) 이상인 채로 착지하면 탈락.
    판정 주체 = 소유자 보고 + 서버: 접지는 소유자 클라만 정확히 알므로(소유자 권위 이동) 착지 시
    `ReportFallServerRpc(fallHeight)` 로 보고(`fallReportMin`=3 이상만)하고 서버가 임계 비교.
  - 탈락 확정 시 서버가 **탈락 위치(월드 좌표)를 통계에 기록**(`DeathEvents`) — 추후 하이라이트의
    "발판 영역 내 단기 킬 집중" 판정 입력.
- **탈락:** 그 라운드 동안 행동 불가. 본체 렌더러·콜라이더 off + **머리 위 이름표도 숨김**(GameHUD 가 `Dead` 로 분기 — 시체 오브젝트는 탈락 위치에 남으므로 이름표를 끄지 않으면 허공에 뜬다, 2026-07-12 수정).
  - 서버가 해당 소유자에게 `EliminatedRpc` 전송 → 소유 클라는 입력 잠금, 본체는 사망 지점에 정지(렌더러 off, 콜라이더 off: 좁은 발판에서 시체가 길을 막지 않게).
  - **관전 모드로 전환:** 카메라가 생존 중인 다른 플레이어를 따라다닌다(좌클릭으로 대상 순환). HUD 에 "탈락, 관전 중: P2" 표시.
  - 부활 없음(라운드 종료 시 일괄 복귀).

## 5. 준비 페이즈 (구조물 설치)

- **설치 개수(라운드별 지급, 이월 없음):**

| 시점 | 보이는 구조물 | 보이지 않는 구조물 |
|---|---|---|
| 라운드 1 시작 전 | 3 | 1 |
| 라운드 2 시작 전 | 2 | 2 |
| 라운드 3 시작 전 | 1 | 3 |

  (시리얼라이즈 배열 `visibleAllowance = {3,2,1}`, `invisibleAllowance = {1,2,3}`)
- **설치 방법:** 위쪽 발판에는 지상에서 설치할 수 없으므로 PREP 동안 **자유 비행 고스트 카메라**로 전환한다: WASD 는 x/z 수평 이동 전용(시선 상하와 무관), Space 상승 / Shift 하강, 마우스룩은 시선 회전만 담당. 이 조작으로 이동 범위(MovementBounds) 안을 날아다니며 좌클릭으로 설치(본체는 시작 섬 스폰 위치에 잠금).
- **설치 모드(Q 토글, 2026-07-12 추가):**
  - **표면 모드(기본)** = 화면 중앙 조준점 레이캐스트 히트점에 설치. **설치된 구조물을 조준하면 그 구조물 AABB 의 윗면으로 자동 스냅** — 옆면을 가져다 대도 위로 올라간다(구조물을 발판처럼 쌓는 플레이 지원).
  - **공중 모드** = 카메라 정면 거리 d 지점에 설치(지면 불필요). **휠 위 = 멀어짐 / 휠 아래 = 다가옴**(`_airDistanceStep`, 범위 `_airDistanceMin~Max`). 전환 순간 d 를 현재 조준점 거리로 초기화해 프리뷰가 튀지 않는다. 매 PREP 은 표면 모드로 시작.
- 종류 선택: 숫자키 1(벽)/2(원기둥) = 보이는 구조물, 3 = 보이지 않는 구조물. R 키로 90도 회전. 남은 개수는 PREP 패널에 종류별 표시.
- **설치 기준점:** 설치점에 프리팹의 **바닥**(피벗이 아니라 렌더 바운즈 최저점)이 닿도록 스폰한다(`Structure.BottomOffset`). 지면/발판 위에 설치해도 땅에 박히지 않는다. 서버 스폰과 클라 블루프린트가 같은 계산을 공유.
- **서버 검증:** PREP 페이즈인지 / 해당 종류 잔여 개수 / **설치 허용 범위 안 + 시작 섬 영역 밖**(`IsPlacementAllowed`, 2026-07-12 개편) / **설치물끼리 관통 금지**(`Structure.OverlapScan`, 관통 허용 오차 `_overlapTolerance` 기본 0.05 — **접촉·면 맞대기는 허용**하므로 구조물 위에 쌓기 가능). 통과 시 서버가 `NetworkObject.Spawn()`(전 클라 복제), 실패는 조용히 거부. 랜덤 발판(레인)과의 겹침은 허용한다: 발판에 끼워/걸쳐 설치하는 것이 게임적으로 유효하다.
- **함정 비노출 겹침 처리(2026-07-12):** 클라 예비검증(블루프린트 초록/빨강)은 **"나에게 보이는" 설치물만** 검사한다 — 타인의 보이지 않는 구조물은 프리뷰를 빨갛게 만들지 않는다(가져다 대는 것만으로 함정 위치가 드러나면 안 됨). 그 자리에 실제로 설치를 시도하면 서버가 거부하고 요청자에게만 `DenyPlacementByTrapRpc` → **"이 근처에 함정이 있습니다!"** 문구를 잠깐 표시(잔여 개수 소모 없음). 따라서 초록 = "내가 아는 한 설치 가능"이며, 함정 스캔에는 실질 비용이 있다(함정이 없으면 그대로 설치돼 개수가 소모되므로). 또한 **조준 레이도 숨겨진 구조물을 통과**한다(`Structure.IsHiddenFromLocal`, 렌더러와 동일 판정) — 통과시키지 않으면 블루프린트가 함정 표면에 얹히거나 윗면 스냅이 작동해 설치 시도 없이 위치가 드러난다.
- 배치 프리뷰(**블루프린트**): 실제 구조물 프리팹을 인스턴스화해 물리/네트워크/로직을 제거하고 렌더러만 남긴 반투명 복제본 -> 실물과 형태 1:1 일치, 새 구조물은 프리팹만 연결하면 자동 지원. 설치점에 표시하며 설치 가능 초록 / 불가 빨강. **선택한 종류의 잔여 개수가 0 이면 블루프린트 자체를 표시하지 않는다.**

## 6. 구조물 가시성 규칙 (핵심 표)

| 상황 | 보이는 구조물 | 보이지 않는 구조물 |
|---|---|---|
| PREP, 설치자 본인 | 보임 | **보임(반투명)** |
| PREP, 타인 | 보임 | 안 보임 |
| PLAY/HIGHLIGHT/RESULT, 전원 | 보임 | 안 보임(설치자 포함) |
| 플레이어와 상호작용(충돌) 발생 시 | (항상 보임) | **`revealDuration`(2초) 동안 전원에게 보임** |

- **콜라이더는 항상 전원에게 동일하게 켜져 있다**(물리 공정성 유지, 기존 Ghost 규약 계승). 가시성은 렌더러만 분기.
- 설치자 본인 반투명(PREP): 전용 머티리얼(`_ownerPreviewMaterial`)이 없으면 원본 머티리얼을 `_ownerPreviewAlpha`(0.45)로 낮춰 표시 — "내가 둔 투명 구조물"임을 원본과 구분할 수 있다(2026-07-12).
- **스폰 깜빡임 방지(2026-07-12):** `Type` 은 서버가 Spawn 직후에 쓰므로 타 클라에는 첫 몇 프레임 Wall(0)로 도착한다. `_renderers` 가 할당된 프리팹(= 보이지 않는 구조물)은 `OnNetworkSpawn` 에서 렌더러를 즉시 꺼 "기본 숨김"으로 시작 → 투명 구조물이 설치 순간 타인에게 잠깐 보이던 문제 제거.
- 상호작용 감지: 충돌한 플레이어의 소유 클라가 `OnControllerColliderHit` 으로 감지 → `RevealStructureServerRpc(구조물 참조)` → 서버가 구조물의 `NetworkVariable<double> RevealUntil = ServerTime + revealDuration` 기록 → 전 클라가 `ServerTime < RevealUntil` 이면 렌더링. 연속 충돌은 시간 연장.
- 공개/숨김 전환은 **알파 페이드**: 페이드 인 `_fadeInDuration`(0.15초) / 페이드 아웃 `_fadeOutDuration`(0.6초). 원본은 불투명 머티리얼이므로 페이드 중에만 런타임 투명 전환 복제본을 쓰고, 완전 공개 시 원본으로 복귀. 프리팹의 렌더러 배열만 채우면 이후 추가되는 보이지 않는 구조물에도 자동 적용(URP Lit 계열 머티리얼 권장). 늦은 합류자는 페이드 없이 현 상태 즉시 반영.
- 기존 `Obstacle.Ghost`(설치자에게 상시 반투명) 규칙은 이 규칙으로 **대체**된다.

## 7. 점수 (※ 임시 로직 — 점수 계산 방식 전면 개편 예정, 2026-07-12 결정)

- 점수 공식은 `MatchScoring.RoundScore`(MatchScoring.cs) **한 곳에만** 존재한다. 설정(`ScoringConfig`: 만점·높이->점수 곡선 `heightCurve`·순위 보너스)은 MatchManager 인스펙터에서 조정 — 곡선을 바꾸면 코드 수정 없이 높이 구간별 가치를 재조정할 수 있다. 새 점수 요소 추가 = ScoringConfig 필드 + RoundPerformance 입력 + RoundScore 항 하나.
- **라운드 점수 = 높이 점수 + 순위 보너스** (호스트 계산, 3라운드 누적):
  - 높이 점수 = (라운드 **종료 시점**의 플레이어 y / `mapHeight`) x `heightScoreMax`(700). 최고점(peak)이 아니라 **종료 시점 높이**라서 마지막까지 안전하게 서 있는 위치가 중요하다. 탈락자는 사망 지점 높이로 계산.
  - 플레이어 높이(HUD 표시·채점·정상 도달 판정)는 **발끝(캡슐 바닥) 기준**으로 계산한다(`PlayerController.FootY`).
  - 순위 보너스: 종료 시점 높이 내림차순 순위에 `rankBonus = {300, 200, 100}` (4위부터 0). 동률이면 그 높이에 먼저 도달한 쪽이 위.
- 정상(y ≥ mapHeight) 도달 시 높이 점수 만점 고정. 첫 도달자가 나오면 잔여 타이머를 `finishGrace`(15초)로 단축(나머지도 기회는 가짐).
- 라운드 종료 조건: 타이머 만료 / 전원 사망 / 정상 도달 후 유예 종료.
- 매치 승자 = 3라운드 누적 총점 1위. 기존 `NetworkList<RoundResult>` 복제 구조 재사용(Topic 필드는 0 고정).

## 11. 결정 사항 (명세 작성자가 정함, 조정 가능)

1. 준비 페이즈 설치는 **자유 비행 카메라**로: 높이 50 타워 특성상 지상 설치만으로는 위쪽 배치가 불가능해서. 본체는 바닥에 잠금.
2. 관전은 **생존자 추적 카메라**(좌클릭 순환)로: "다른 사람의 화면 관전" 요구를 3인칭 추적으로 해석(1인칭 화면 미러링은 네트워크로 불가).
3. 탈락자 시체는 렌더러·콜라이더 off: 좁은 발판 길막 방지.
4. 종료 시점 높이(peak 아님)로 점수 계산, 사망자는 사망 지점 높이.
5. 상호작용 공개 시간 기본 2초, 연속 충돌 시 연장.
6. 타이머 기본값: PREP 60초 / PLAY 180초(밸런싱 대상).
7. 랜덤 구조물은 결정론(시드 복제) 생성: 150개 NetworkObject 스폰 회피, late-join 대응.

## 12. LAN 로비/대기방 (2026-07-12 추가)

시연용 사용자 흐름: **타이틀 → 방 만들기/방 참가 → 대기방 → 게임 → 결과 → 대기방 복귀**.

|---|---|---|
| `Net/ConnectionService.cs` | NetworkManager 오브젝트에 Add Component | 연결 단일 창구: StartHost/StartClient/Shutdown, 포트 자동 탐색(ActivePort), listen 0.0.0.0, 표시용 LAN 사설 IPv4 탐색(루프백 제외, 192.168>10>172.16~31 우선), 접속 승인. **[Relay 전환 지점]** = CreateRoom/JoinRoom 내부 |
| `Match/LobbyManager.cs` | MatchManager 오브젝트에 Add Component(같은 NetworkObject 공유) | 서버 권위 로비 상태: `NetworkList<LobbyPlayerState>{ClientId, Name(FixedString64), Ready, IsHost}` + 이름/준비/시작 RPC. LOBBY 재진입을 스스로 감지해 준비 초기화(의존 방향 Lobby→Match 한쪽). 없으면 MatchManager 가 기존처럼 lobbyDuration 자동 시작 |
| `UI/LobbyUI.cs` | 전용 GameObject "LobbyUI"에 Add Component | 화면 상태 머신: Title/Join/Connecting/Waiting/InGame. 상태는 매 프레임 NGO 에서 유도(이벤트 구독 없음), 접속 타임아웃(직렬화), 거절 사유(DisconnectReason) 표시. 정식 UI 는 추후 uGUI 프리팹 교체 대상 |
| `Net/NetworkBootstrap.cs` | (기존 씬 컴포넌트 유지) | 디버그 즉시 접속 패널로 축소(기본 숨김 `_showDebugGui=false`), ConnectionService 위임 |

게임 시작 연결점: `LobbyManager.StartGameServerRpc`(검증) → `MatchManager.StartMatchFromLobby()`(FSM 진입).

### 규칙

- 로비 상태는 서버만 쓴다. 클라는 자기 준비 상태만 RPC 로 요청(다른 플레이어 변경 불가).
- 게임 시작 조건(서버 재검증): LOBBY + 요청자가 호스트 + `_minPlayers`(기본 2) 이상 + 전원 준비.
- 신규 접속 정책(ConnectionApproval): LOBBY 에서만 허용. PREP 이후 = "게임이 이미 진행 중입니다",
  `_maxPlayers`(기본 4) 초과 = "방이 가득 찼습니다" 로 거절(사유는 클라 DisconnectReason 으로 전달).
- 대기방 복귀 시 `NetworkManager.Shutdown()` 금지 — 연결 종료는 방 나가기/호스트 종료뿐.
- 닉네임: 앞뒤 공백 제거, 빈 값 기본 이름, 12자 제한, 중복 시 `#클라번호` 접미(서버 정제).
  표시 접근점은 `PlayerPalette.NameFor`(로비 닉네임 우선, P{n} 폴백) 하나로 유지.
- 화면 겹침 방지: LOBBY 중 GameHUD 는 전부 숨김, PlayerController 는 커서 해제·입력 잠금(중력만).
