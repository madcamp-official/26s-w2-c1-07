# 완성도 강화 계획 (Day 7 이후)

> 코어 게임 루프(접속·맵·매치 FSM·준비 페이즈·점수·HUD·조준 시점)는 완성 상태다.
> 하지만 실제로 에디터에서 플레이해보면 "게임처럼" 느껴지지 않는다 — 캐릭터가 캡슐이고,
> 사운드·파티클이 전혀 없고(코드베이스 전체에 `AudioSource`/`ParticleSystem`/`Animator` 사용처 0곳),
> 장애물은 3종뿐이고, 배포는 로컬 IP 접속뿐이다.
> 이 문서는 "골든 패스가 돈다"에서 "플레이하면 재미있고 그럴듯하다"로 가기 위한 남은 작업을 정리한다.
>
> **갱신(커밋 `9a00bf54`):** 맵이 절차 생성 지형에서 **폴가이즈풍 캔디 코스**(`Map_FallGuys`, ProBuilder 수작업)로
> 전면 교체됐다. 출발 광장 → 징검다리 → 경사로 → 하이 덱(범퍼) → 메인 덱+나선 타워(높이 모드용, 최고 12)
> → 좁은 다리 3열 → 슬라롬 벽 → 골 광장 구성이며, 스폰/장애물 배치 검증도 `MapSurface.cs`(Ground 레이어
> 레이캐스트)로 맵-독립적으로 바뀌었다. **1절(맵 만들기)은 원래 계획보다 앞서 진행됐으므로 아래 내용을
> "1차 완료 + 다음 폴리싱"으로 갱신했다.**

## 1. 맵 만들기 (1차 완료 — 다음 폴리싱)

**현황:** `Map_FallGuys`(ProBuilder 수작업, 씬 정적 지오메트리)로 절차 지형을 대체 완료. 출발 광장·징검다리·경사로·범퍼 하이 덱·메인 덱+나선 타워(높이 모드용)·좁은 다리 3열·슬라롬 벽·골 광장까지 시그니처 구간이 이미 갖춰져 있고, 캔디 파스텔 머티리얼 8종과 슬라임 바다(시각용 낙사 연출)까지 적용됨. 스폰/장애물 배치 판정은 `MapSurface.cs`가 `Ground` 레이어 레이캐스트로 처리해서 맵을 더 손봐도 코드 수정이 필요 없다. `docs/개발가이드.md` 3절에 편집 규칙(Ground 레이어 유지, 점프 한계 1.5/틈 3 이하, 높이 모드 만점 y=12)이 정리돼 있다.

| 작업 | 설명 | 우선순위 |
|---|---|---|
| 분위기 연출 | 스카이박스(그라데이션), 라이팅 색, URP Volume 포스트프로세싱(블룸+비네트) — 개발가이드에서도 "가성비가 크다"고 이미 지목한 다음 항목 | 필수 |
| 플레이테스트 밸런싱 | 실제 8명 내외로 접속해 징검다리·좁은 다리·슬라롬 구간 난이도, 나선 타워 높이 모드 체감, 골든 패스 완주 소요 시간 점검 | 필수 |
| 스폰 공간 재확인 | 출발 광장(X 약 -7~9)이 목표 동시 접속 인원(8~16명)을 실제로 수용하는지 겹침 없이 확인 | 필수 |
| 추가 시그니처 구간 | 여유가 있으면 라운드 반복 플레이 시 지루하지 않도록 우회 경로·숏컷 등 변주 추가 | 권장 |
| 장애물 배치 구역 시각화 | PREP 페이즈에서 배치 가능 구역을 바닥 마킹 등으로 표시 | 선택 |
| README 참고자료 표 최신화 | README의 "KCLOUD VM 환경별 주의사항" 등 곁가지 표는 그대로 두되, 이미 처리된 맵 관련 문서 갱신(완료됨, 재확인만) | 완료 확인만 |

| 작업 | 설명 | 우선순위 |

## 4. 전반적으로 게임처럼 만들기 / 플레이 필(feel)

**목표:** `docs/개발가이드.md` 4절에 이미 착수 순서가 정해져 있다("걷기/점프 자연스러운 캐릭터 → 색 구분 → 이모트"). 여기에 파티클·사운드·충돌 피드백을 더해 "허접한 그래픽 + 웃긴 장면"이라는 기획 의도(기획서 7.1절)를 실제로 살린다.

| 작업 | 설명 | 우선순위 |
|---|---|---|
| 저폴리 캐릭터 모델 부착 | 무료 에셋 또는 직접 제작 → `Player.prefab` 자식으로 추가(루트의 CharacterController/NetworkObject는 유지). 개발가이드 순서 그대로 | 필수 (현재 캡슐 상태 추정 — 데모 임팩트에 가장 크게 영향) |
| 이동 애니메이션 | `Animator` 추가, `PlayerController`가 이미 계산하는 이동 속도로 Idle/Run/Jump 파라미터 구동 | 필수 |
| 색 구분 적용 | `PlayerTint`(이미 구현됨, 미부착 상태로 추정) 컴포넌트를 `Player.prefab` 루트에 추가만 하면 됨 | 필수 (구현은 끝나 있어서 작업량 적음) |
| 낙사 파티클 + 사운드 | 강물 낙사 시 물보라 파티클 + 첨벙 사운드. 기획서에서 "웃긴 장면" 재미의 핵심 요소로 명시됨 | 필수 |

| 배경음악 | 페이즈별(로비/준비/플레이/결과) 인텐시티 변화 | 권장 |
| 충돌/착지 피드백 | 착지 스쿼시, 장애물 충돌·낙사 시 짧은 스크린 셰이크 | 선택 |
| 사망/부활 피드백 | 현재 로직상 `PendingRespawn`/`RespawnAtServerTime`으로 서버가 부활을 관리하는데(MatchTypes.cs), 클라측 화면 페이드 등 시각 피드백이 있는지 확인 후 없으면 추가 | 권장 |

## 5. 서버 배포

**목표:** 데모 환경에서 실제로 여러 사람이 접속 가능하게 만든다. 먼저 **배포 방식을 결정**하는 것이 이 항목의 첫 작업이며, 나머지 폴리싱 작업의 테스트 방식에도 영향을 준다.

### 방식 비교

| 방식 | 작업량 | 장점 | 단점 |
|---|---|---|---|
| A. 현재 그대로 (LAN 직접 IP) | 없음 | 이미 동작함 | 발표장이 같은 네트워크가 아니면 접속 불가 |
| B. Unity Relay (Unity Gaming Services) | 적음 | NGO가 UTP+Relay 네이티브 지원, Steam 없이 NAT 우회, 무료 티어 | Unity Dashboard 프로젝트 연결 필요 |
| C. KCLOUD 헤드리스 전용 서버 | 많음 | 상시 접속 가능, README 구현명세서의 "전용 서버(DGS)" 방향과 일치 | KCLOUD VM은 기본 22/80/443 포트만 허용 — 게임 UDP 트래픽 위해 보안그룹에서 커스텀 포트 개방 필요 (week2 README 참고자료의 "KCLOUD VM 환경별 주의사항" 표 참고) |
| D. Facepunch/Steam 릴레이 | 큼 | 원래 설계 목표, 출시 대비 | Steamworks 파트너 계정·App ID·`steam_appid.txt` 등 인프라 필요, 데모엔 과함 |

**권장:** B(Unity Relay)를 데모용 기본으로, 시간 여유가 있으면 C를 병행. D는 "선택"으로 계속 보류(README 구현명세서에도 이미 이렇게 방향이 잡혀 있었음 — 단지 README 본문 텍스트만 "필수"로 오기돼 있으니 정정 필요).

### 세부 작업 (B 기준)

| 작업 | 설명 | 우선순위 |
|---|---|---|
| Unity Gaming Services 프로젝트 연결 | Unity Dashboard에서 프로젝트 생성/연결(무료) | 필수 |
| 패키지 설치 | `com.unity.services.relay`, `com.unity.services.authentication` | 필수 |
| 트랜스포트 교체 지점 활용 | `NetworkBootstrap.ApplyConnectionData()` 한 곳만 Relay 할당(호스트: `CreateAllocation`, 클라: `JoinAllocation`)으로 교체 — 설계상 이미 이 메서드로 격리돼 있음(코드 주석에 명시) | 필수 |
| Join Code UI | 호스트가 발급받은 코드를 표시, 클라가 입력하는 화면(2절 로비 UI 작업과 함께 처리) | 필수 |
| 실제 이종 네트워크 테스트 | 팀원 각자 다른 네트워크(집/카페 등)에서 실제 접속 검증 | 필수 |
| (병행 시) KCLOUD 헤드리스 서버 | Linux 헤드리스 빌드 + 보안그룹 UDP 포트 개방 + 상시 기동/자동 재시작 | 선택 |

## 6. 실행 순서 제안 (2인 팀 기준)

1. **(공동, 가장 먼저) 서버 배포 방식 결정** — LAN으로 충분한 데모라면 5절 작업 전체를 생략할 수 있으므로, 이걸 먼저 확정해야 이후 테스트 방식(로컬 2빌드 vs 실제 원격 접속)이 정해진다.
2. **콘텐츠 트랙**: 맵 분위기 연출·밸런싱(1절, 구조는 이미 완료) + 장애물 신규 타입(3절)
3. **필(feel) 트랙**: 캐릭터 모델·애니메이션·파티클·사운드(4절) + UI 폴리싱(2절)
4. **통합**: 새 맵 기준 전체 플레이테스트 → 밸런싱(장애물 개수/점수 비율 등) → Relay/서버 연동 최종 검증
5. **마무리**: Windows 빌드, 시연 영상, README 회고(Keep/Problem/Try, 팀원 소감) 작성, README "개발 일정" 표를 실제 진행 상황(Day 7까지 완료)에 맞게 갱신
