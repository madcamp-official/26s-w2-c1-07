# 완성도 강화 계획 (Day 7 이후)

> 코어 게임 루프(접속·맵·매치 FSM·준비 페이즈·점수·HUD·조준 시점)는 완성 상태다.
> 하지만 실제로 에디터에서 플레이해보면 "게임처럼" 느껴지지 않는다. 캐릭터가 캡슐이고,
> 사운드·파티클이 전혀 없고(코드베이스 전체에 `AudioSource`/`ParticleSystem`/`Animator` 사용처 0곳),
> 장애물은 3종뿐이고, 배포는 로컬 IP 접속뿐이다.
> 이 문서는 "골든 패스가 돈다"에서 "플레이하면 재미있고 그럴듯하다"로 가기 위한 남은 작업을 정리한다.
>
> **갱신(커밋 `9a00bf54`):** 맵이 절차 생성 지형에서 **폴가이즈풍 캔디 코스**(`Map_FallGuys`, 코드 배치 정적 지오메트리)로
> 전면 교체됐다. 출발 광장 → 징검다리 → 경사로 → 하이 덱(범퍼) → 메인 덱+나선 타워(높이 모드용, 최고 12)
> → 좁은 다리 3열 → 슬라롬 벽 → 골 광장 구성이며, 스폰/장애물 배치 검증도 `MapSurface.cs`(Ground 레이어
> 레이캐스트)로 맵-독립적으로 바뀌었다. 1절(맵 만들기)은 "1차 완료 + 다음 폴리싱"으로 갱신.
>
> **갱신 2(상세화):** 각 작업을 기능명세서 수준(구현 방법·관련 파일·완료 기준)으로 상세화했다.
> 문서가 "확인 필요"로 남겼던 항목은 코드에서 검증해 결과를 0절에 적었고, 검증 중 발견한
> 누락 작업(스폰 슬롯 공식, 생존 탈락자 방치, 연결 끊김 UX 등)을 (신규) 표시로 추가했다.

## 1. 맵 만들기 (1차 완료, 다음 폴리싱)

**현황:** `Map_FallGuys`로 절차 지형 대체 완료. 편집 규칙은 `docs/개발가이드.md` 3절(Ground 레이어 유지, 점프 한계 1.5/틈 3 이하, 높이 모드 만점 y=12).

| 작업 | 우선순위 |
|---|---|
| 1-1 분위기 연출 | 필수 |
| 1-2 플레이테스트 밸런싱 | 필수 |
| 1-3 스폰 슬롯 공식 수정 (신규) | 필수 |
| 1-4 추가 시그니처 구간 | 권장 |
| 1-5 장애물 배치 구역 시각화 | 선택 |

### 1-1 분위기 연출
- **구현:** (a) `Window > Rendering > Lighting > Environment`에서 Skybox Material을 그라데이션 계열로 교체(에셋 없이 하려면 `Skybox/Procedural` 셰이더 머티리얼 생성 후 Sky Tint 하늘색·Exposure 1.3). (b) Directional Light 색 `(1.0, 0.96, 0.90)`, Intensity 1.1, 회전 (50, -30, 0). (c) 빈 오브젝트 `PostFX` + `Volume`(Global) 컴포넌트 → 프로파일 생성 → Bloom(Intensity 0.3, Threshold 1.1) + Vignette(Intensity 0.25) + Tonemapping(ACES).
- **파일:** 씬(test.unity), `Assets/Materials/Map/M_Sky.mat`(신규), Volume 프로파일 에셋.
- **완료 기준:** 플레이 스크린샷에서 파스텔 톤이 유지되면서 밝은 부분에 블룸이 보이고, 이전 스크린샷 대비 "회색 하늘" 느낌이 사라짐.

### 1-2 플레이테스트 밸런싱
- **구현:** `MatchManager`의 `fastMode`를 끄고 풀타이머(PLAY 45초)로 2~4인 실측. 관찰 지표를 정해 기록한다: 완주 소요(목표 25~35초), 징검다리/좁은 다리 구간별 낙사 횟수, 높이 모드에서 타워 정상(y=12) 도달자 수(목표: 라운드당 1~2명만 도달하는 난이도), 생존 모드에서 장애물로 인한 낙사 비율.
- **조정 노브:** 틈 간격(현재 2~3), 다리 폭(현재 2.2), 슬라롬 벽 간격, `_killRadius`(4), `KILL_SCORE`(100), `OBSTACLE_CAP`(300), `HEIGHT_MAX`(12), PLAY 시간(45초).
- **완료 기준:** 지표 기록표가 채워지고, 조정 후 "레이스에서 전원 완주 실패" 또는 "전원 무낙사 완주" 같은 극단 케이스가 사라짐.

### 1-3 스폰 슬롯 공식 수정 (신규)
- **문제:** `MatchManager.SpawnSlot(index)`가 `x = (index - 3) * spawnSpacingX`로 8인 가정 하드코딩이라, 9인부터 출발 광장(X ±13) 밖(x≥12)에 스폰돼 슬라임으로 떨어진다.
- **구현:** 시그니처를 `SpawnSlot(int index, int total)`로 바꾸고 `x = (index - (total - 1) * 0.5f) * spawnSpacingX; x = Mathf.Clamp(x, -11f, 11f);`. 호출부(`BeginPlay`, `RescueFallenPlayers`)에서 `NetworkManager.ConnectedClientsList.Count`를 전달. 클램프로 겹침이 생기는 17인 이상은 지원 범위 밖(README의 MVP 8~16명 기준 유지).
- **완료 기준:** MPPM 가상 플레이어 포함 9인 이상 접속 시 전원이 광장 위에 스폰(낙하 구조 로그 0회).

### 1-4 추가 시그니처 구간 (권장)
- **구현:** 기존 구간 사이에 우회/숏컷 하나를 추가한다. 예: 메인 덱(z -6~4) 왼쪽(x -16 부근)에 폭 1.5의 "리스크 숏컷" 다리(짧지만 좁음), 슬라롬 구간 위(y 3)에 점프로만 진입 가능한 고가 루트. 규칙은 개발가이드 3절 그대로(Ground 레이어, 틈 3 이하).
- **완료 기준:** 플레이테스트에서 두 경로의 선택 비율이 갈리고(대략 7:3 이상 쏠리지 않음), 숏컷 이용 시 1~3초 이득.

### 1-5 장애물 배치 구역 시각화 (선택)
- **구현:** PREP 진입 시에만 켜지는 반투명 경계 표시. 코스 경계(z=START_Z와 z=GOAL_Z)에 반투명 벽 Quad 2장(콜라이더 없음, `Ground` 레이어 아님)을 두고 `MatchManager` 페이즈 변화에 따라 클라 로컬로 SetActive. 배치 불가 판정 자체는 서버가 하므로 시각 안내만.
- **완료 기준:** PREP에서만 경계가 보이고, 경계 밖 클릭이 (기존처럼) 조용히 거부될 때 이유를 눈으로 알 수 있음.

## 2. UI 개선

**목표:** 기능은 이미 다 있는 `GameHUD.cs`에 전환감·피드백·사운드를 입혀 "완성된 화면"처럼 보이게 한다.

| 작업 | 우선순위 |
|---|---|
| 2-1 로비/접속 화면 정식화 | 필수 |
| 2-2 룰렛 연출 강화 | 필수 |
| 2-3 사운드 이펙트 | 필수 (4절 4-6 SfxManager로 통합 구현) |
| 2-4 페이즈 전환 페이드 | 권장 |
| 2-5 스코어보드/하이라이트/결과 연출 | 권장 |
| 2-6 색·폰트 시스템 통일 | 권장 |
| 2-7 디버그 GUI 정리 | 필수 |
| 2-8 미니맵/방향 인디케이터 | 선택 |

### 2-1 로비/접속 화면 정식화
- **구현:** 신규 `Assets/Scripts/UI/TitleScreen.cs`(GameHUD와 같은 코드 생성 uGUI 방식). 화면 구성: 중앙 타이틀 "룰렛 파티"(대형 텍스트+그림자), 그 아래 IP 입력 `InputField`(uGUI 레거시, TMP 불필요), [방 만들기] [참가하기] 버튼 2개, 하단에 조작법 요약 1줄.
  - **입력 처리 주의:** GameHUD는 표시 전용이라 Raycaster가 없다. TitleScreen 캔버스에는 `GraphicRaycaster`를 붙이고, 씬에 `EventSystem` + **`InputSystemUIInputModule`**(신 Input System 전용, 구 StandaloneInputModule은 동작 안 함)을 자동 생성한다.
  - **로직 재사용:** 버튼은 기존 `NetworkBootstrap`의 Host/Client 경로를 그대로 호출(public 메서드로 노출: `StartHostGame()`, `StartClientGame(string ip)`). 트랜스포트 설정은 계속 `ApplyConnectionData()` 한 곳.
  - **상태 전환:** 미접속=타이틀 표시, `OnClientStarted`/`OnServerStarted`=타이틀 숨김, 종료/끊김=타이틀 복귀. 커서는 미스폰 상태에서 원래 자유이므로 추가 처리 불필요.
  - **기존 OnGUI:** `NetworkBootstrap`에 `showDevGui`(기본 false) 직렬화 필드를 추가해 디버그용으로 보존.
- **완료 기준:** 에디터 Play 직후 디버그 박스 대신 타이틀 화면이 보이고, 마우스로 IP 입력·접속이 되고, Shutdown/끊김 시 타이틀로 복귀.

### 2-2 룰렛 연출 강화
- **구현:** `GameHUD.RenderRoulette()` 교체. (a) 텍스트 3후보를 세로 릴(휠)로 배치해 `anchoredPosition`을 감속 커브(`1-Mathf.Pow(1-t,3)`)로 스크롤, 최종적으로 실제 `Topic`에 정착(현 로직의 "결정된 주제에 정착" 규약 유지). (b) 릴 인덱스가 바뀔 때마다 틱 사운드(`SfxManager.Play("roulette_tick")`). (c) 정착 순간 당첨 텍스트 스케일 펀치(1→1.25→1, 0.3초)와 색 플래시, 팡파레 짧게.
- **완료 기준:** 룰렛 5초 동안 릴이 돌다 감속해 멈추고, 멈춘 항목 = 실제 라운드 주제(불일치 0회), 틱/팡파레 사운드 재생.

### 2-3 사운드 이펙트: 4절 4-6(SfxManager) 명세로 통합. UI 훅 지점만 여기서 정의: 버튼 클릭(TitleScreen), 투표 선택(PrepClientUI.CastVote), 배치 성공(내 소유 Obstacle 스폰 감지), 룰렛 틱/당첨, 카운트다운 마지막 3초 비프(GameHUD 남은시간), 라운드 종료 휘슬, 챔피언 팡파레(Result 진입).

### 2-4 페이즈 전환 페이드 (권장)
- **구현:** `GameHUD`에 전체 화면 검정 `Image` + `CanvasGroup` 1장 추가. `RefreshDynamic`에서 직전 프레임 페이즈와 달라진 순간 `_fadeT=1`로 세팅하고 매 프레임 `_fadeT -= Time.deltaTime/0.25f`, `alpha = _fadeT` (0.25초 페이드인). 코루틴 불필요, 네트워크 무관(클라 로컬).
- **완료 기준:** 모든 페이즈 전환에서 짧은 페이드가 보이고 HUD 조작(없음)에 영향 없음.

### 2-5 스코어보드/하이라이트/결과 연출 (권장)
- **구현:** (a) 스코어보드 순위 변동 시 행 `anchoredPosition`을 목표값으로 `Mathf.Lerp`(스냅 대신 0.2초 슬라이드). (b) 하이라이트 카드 등장 시 스케일 0.9→1 + 알파 0→1 (0.3초). (c) 결과 화면 챔피언 텍스트에 무지개 색상 순환(`Color.HSVToRGB(Mathf.Repeat(Time.time*0.3f,1),0.6f,1)`) + 행들이 위에서 순차 등장(행당 0.1초 지연).
- **완료 기준:** 라운드 사이 순위 역전 시 행이 미끄러지듯 재배치되고, 결과 화면이 한 번에 뜨지 않고 순차 연출됨.

### 2-6 색·폰트 시스템 통일 (권장)
- **구현:** `PlayerPalette.cs`에 UI 상수 추가(또는 신규 `UIStyle.cs`): 본문/제목 폰트 크기, 패널 배경색(현 검정 50%), 강조색, 여백 상수를 한 곳으로. `GameHUD`/`TitleScreen`/`PrepClientUI`(OnGUI 유지 시 색만)가 이 상수를 참조.
- **완료 기준:** 색·크기 하드코딩이 UIStyle 한 파일로 모이고 화면 간 톤이 통일.

### 2-7 디버그 GUI 정리 (필수)
- **구현:** `MatchManager.showDebugGui` 기본값 false로 변경(인스펙터에서 켜서 디버깅), `NetworkBootstrap` OnGUI는 2-1의 `showDevGui`로 숨김. PREP 정보 패널(PrepClientUI)은 게임 UI이므로 유지하되 2-6 스타일 적용 대상.
- **완료 기준:** 기본 Play 화면에 개발자 박스가 하나도 안 보이고, 필요 시 인스펙터 체크로 복구 가능.

### 2-8 미니맵/방향 인디케이터 (선택)
- **구현(간단안):** 화면 상단에 코스 진행률 바(내 z를 START_Z~GOAL_Z로 정규화한 0~1)를 표시하고 다른 플레이어 위치를 색점으로 오버레이(레이스 주제일 때만). 미니맵 카메라 방식은 과함.

## 3. 장애물 추가

**목표:** 이동형 장애물·이로운 배치물을 추가해 준비 페이즈의 심리전을 풍부하게 한다. `Obstacle.cs`의 기존 패턴(콜라이더는 항상 켜짐=공정, 렌더러만 클라별 분기)을 그대로 따른다.

| 작업 | 우선순위 |
|---|---|
| 3-1 트램펄린 추가 | 권장 (재미 임팩트 큼) |
| 3-2 배치 프리뷰 | 권장 |
| 3-3 키 매핑 재편 | 신규 타입 추가 시 필수 |
| 3-4 회전 장애물 | 선택 |
| 배치 개수 제한 | 검증 완료: 서버 로직 이미 존재(0절). 신규 타입별 개수 밸런스만 1-2에서 함께 조정 |

### 3-1 트램펄린
- **권위 모델 (원안 정정):** 원안의 "서버 판정 후 TeleportPlayerRpc로 보정"은 이 프로젝트의 이동 계약과 맞지 않다. 이동은 **소유자 권위**(ClientNetworkTransform)이므로, 서버가 개입하면 왕복 지연으로 바운스가 늦게 터진다. **올바른 방식: 소유 클라 로컬 반응.** 장애물 콜라이더는 모든 클라에 동일하게 존재하므로(프로젝트 공정성 규약) 소유자 로컬에서 튕겨도 전원에게 동일 조건이고, 위치는 기존 경로로 복제된다. 서버 개입 불필요.
- **구현:**
  1. `MatchTypes.cs`의... (정확히는 `Obstacle.cs`) `ObstacleType`에 `Trampoline = 3` 추가.
  2. 프리팹 `Assets/Prefabs/Obstacle_Trampoline.prefab`: 루트(NetworkObject+Obstacle) + 자식 Cylinder(scale 1.6, 0.25, 1.6, 윗면 민트/스프링 코럴 링), BoxCollider 또는 CapsuleCollider(isTrigger OFF), `ObstaclePrefabs.asset`에 등록 + `MatchManager._trampolinePrefab` 필드 추가와 `PrefabFor()` 분기.
  3. `PlayerController`에 `OnControllerColliderHit(ControllerColliderHit hit)` 추가(소유자만): `hit.normal.y > 0.7`이고 `hit.collider.GetComponentInParent<Obstacle>()?.ObType == Trampoline`이면 `_verticalVelocity = Mathf.Sqrt(2f * bounceHeight * -gravity)`(직렬화 필드 `bounceHeight = 6f`, gravity -20 기준 약 15.5). 재트리거 방지 쿨다운 0.2초.
  4. 피드백: 바운스 순간 `SfxManager.Play("boing")` + 트램펄린 스케일 펀치(모든 클라에서 보이려면 Obstacle에 로컬 감지 추가: 자기 위 플레이어의 y 급상승 감지 또는 생략하고 사운드만 소유자 로컬).
- **밸런스:** 높이 6이면 하이 덱(2.5)과 타워 2~3층을 건너뛸 수 있다. 높이 모드에서 유리하므로 배치 개수 공유 한도(1인 3개) 안에서 자연 균형. 필요 시 종류별 상한(트램펄린 1인 1개)을 `ValidatePlacement`에 추가.
- **완료 기준:** 트램펄린 밟으면 즉시(지연 없이) 튕기고, 원격 클라 화면에서도 그 플레이어가 튀어오르는 게 자연스럽게 보이고(복제 경로), 생존/레이스/높이 3모드 모두에서 동작.

### 3-2 배치 프리뷰 (권장)
- **구현:** `PrepClientUI`에 클라 로컬 고스트 1개(선택 종류의 메시 복제, 콜라이더 제거, 반투명 머티리얼). 매 프레임 화면 중앙 레이캐스트 히트점에 표시하고, 로컬 예비 검증(코스 z 구간 안 + Ground 히트 존재 + 내 남은 개수 > 0)을 통과하면 초록, 아니면 빨강 틴트. 좌클릭 시 기존 `PlaceObstacleServerRpc` 그대로(최종 판정은 여전히 서버, 프리뷰는 안내일 뿐). 종류 변경(1~4키) 시 고스트 메시 교체. PREP 종료 시 고스트 파괴.
- **완료 기준:** 배치 전 위치·종류를 눈으로 확인할 수 있고, 빨강 상태에서 클릭하면 (서버 거부와 일치하게) 안 생김.

### 3-3 키 매핑 재편 (신규 타입 추가 시 필수)
- **문제:** 현재 장애물 1/2/3 + 투표 4/5/6이라 4번째 장애물 키가 투표와 충돌.
- **구현:** 장애물 종류 = `1~5`(예약 포함), 투표 = `8/9/0`으로 이동. 수정 지점: `PrepClientUI.Update()` 키 분기, PREP 패널 안내 문구, `docs/조준_시점_명세서.md` 3.1절 표, README 준비 페이즈 행. 부록 A(키 매핑 레지스트리)를 단일 출처로 삼는다.
- **완료 기준:** 문서 3곳과 실제 키가 일치, 이모트(4절)와도 충돌 없음.

### 3-4 회전 장애물(휩쓰는 바) (선택)
- **구현(원안 개선):** 매 프레임 NetworkVariable 각도 복제 대신 **결정론적 시간 동기화**를 쓴다. 신규 `RotatingBar : NetworkBehaviour`(맵 고정물, 씬 배치 NetworkObject): 서버가 스폰 시 `NetworkVariable<double> _startTime = ServerTime.Time` 1회 기록, 모든 피어가 매 프레임 `angle = speedDeg * (float)(ServerTime.Time - _startTime.Value)`로 회전(대역폭 0, 완전 동기). 밀치기는 트램펄린과 같은 이유로 소유자 로컬: 바에 트리거 자식 콜라이더를 두고 `PlayerController.OnTriggerStay`(소유자만)에서 바 중심 기준 바깥 방향으로 수평 속도 부여.
- **완료 기준:** 두 클라 화면에서 바 각도가 일치(±5도)하고, 맞으면 밀려나며, 호스트 재시작 후에도 동일.

**목표:** "허접한 그래픽 + 웃긴 장면"이라는 기획 의도(기획서 7.1절)를 실제로 살린다. 착수 순서는 개발가이드 4절(캐릭터 → 색 → 이모트) 그대로.

| 작업 | 우선순위 |

| 4-1 저폴리 캐릭터 모델 + 애니메이션 | 필수 |

| 4-3 낙사 파티클 + 사운드 | 필수 |
| 4-4 생존 탈락자 관전 처리 (신규) | 필수 |

### 4-1 저폴리 캐릭터 모델 + 애니메이션
- **에셋 소스(무료, 라이선스 안전):** Quaternius "Ultimate Animated Character Pack"(CC0, Run/Jump/Idle 애니 포함이라 최우선 추천), Kenney "Blocky Characters"(CC0, 애니 없음), Mixamo(무료 자동 리깅+애니, Adobe 계정 필요). 폴가이즈 느낌은 통통한 휴머노이드 1종이면 충분.
- **구현:**
  1. FBX 임포트 → `Player.prefab` 루트의 **자식**으로 추가(루트의 CharacterController/NetworkObject/기존 컴포넌트 유지). 캡슐 MeshRenderer는 비활성/삭제.
  2. 스케일을 CC 캡슐(높이 2, 반지름 0.5)에 맞추고 발이 로컬 y=0에 오게 오프셋.
  3. `Animator Controller`(Assets/Animations/Player.controller): 상태 Idle/Run/Jump, 파라미터 `Speed(float)`, `Grounded(bool)`, `Emote1/Emote2(trigger)`. Idle↔Run은 Speed 0.2 경계 전이, Grounded false면 Jump 상태.
  4. 신규 `Assets/Scripts/Net/PlayerAnimDriver.cs`(모델 자식에 부착, **모든 피어에서 동작해야 원격 플레이어도 애니가 나온다**): 소유자든 원격이든 동일하게 `transform.position` 프레임 차분으로 수평 속도를 계산해 `Speed`에 넣고, `Grounded`는 발밑 0.3m Ground 레이어 레이캐스트로 판정(원격에는 CharacterController 시뮬레이션이 없으므로 CC.isGrounded를 쓰면 안 됨).
- **완료 기준:** 호스트+MPPM 클라 양쪽 화면에서 내 캐릭터와 상대 캐릭터 모두 정지/달리기/점프 애니가 재생되고, 텔레포트(부활) 순간 애니가 튀지 않음(차분 속도에 상한 클램프).

### 4-2 색 구분
- **구현:** `Player.prefab` 루트에 기존 `PlayerTint` 컴포넌트 추가(끝). 모델 부착 후에도 그대로 두면 모델 머티리얼에 색이 입혀진다(MaterialPropertyBlock이라 공유 머티리얼 안전).
- **완료 기준:** 2인 접속 시 서로 다른 색 + 이름표 색과 일치.

### 4-3 낙사 파티클 + 사운드
- **네트워크 설계(중요):** 별도 RPC가 필요 없다. 모든 클라가 모든 플레이어의 위치를 이미 복제받으므로, **각 클라가 로컬로 감시**하면 전원 화면에서 물보라가 터진다.
- **구현:** 신규 `Assets/Scripts/UI/DeathFxWatcher.cs`(씬 오브젝트 또는 GameHUD가 겸임): 매 프레임 스폰된 플레이어 오브젝트들의 y를 추적, 직전 프레임 y ≥ `Course.DEATH_Y`이고 현재 y < `DEATH_Y`인 "하강 교차" 순간에 그 (x, z)의 슬라임 수면 높이(-1.2)에서 물보라 파티클 재생 + `SfxManager.Play("splash")`. 파티클은 코드 생성 `ParticleSystem` 1개를 풀링(burst 40개, 시작색 핑크 `(1, 0.48, 0.72)`, 중력 1, 수명 0.8초, 크기 0.2~0.5). 부활 텔레포트(상승 교차)는 무시.
- **완료 기준:** 내가 떨어져도, 상대가 떨어져도, 양쪽 화면 모두에서 낙하 지점에 물보라+첨벙 소리. RPC/NetworkVariable 추가 0개.

### 4-4 생존 탈락자 관전 처리 (신규, 필수)
- **문제(0절 검증):** 생존 모드 탈락자는 부활 예약이 없어 슬라임 아래로 무한 낙하하며 방치된다(주석은 "관전"이라 쓰여 있으나 실제 처리 없음).
- **구현:**
  1. 맵에 관전대 추가: `SpectatorPerch` 플랫폼 (0, 18, 85), 크기 (10, 1, 6), Ground 레이어. z=85는 배치 가능 구간(GOAL_Z~START_Z) 밖이라 장애물 배치 불가, 코스 전체가 내려다보임.
  2. `MatchManager.SamplePlayers()`의 사망 분기에서 `allowRespawn == false`(생존)일 때 `TeleportOwner(clientId, 관전대 위 슬롯)` 호출(이미 있는 소유자 텔레포트 경로 재사용, 슬롯은 SpawnSlot처럼 index로 벌림).
  3. 탈락 알림: 같은 지점에서 `EliminatedRpc(RpcTarget.Single(clientId))` 1개 추가 → 수신한 소유 클라의 `PlayerController.IsEliminated = true` → `GameHUD`가 이 플래그를 읽어 "탈락! 관전 중" 배너 표시. 다음 `BeginPlay()`의 출발선 텔레포트가 자연 복귀시키므로 라운드 시작 시 `IsEliminated = false` 리셋(같은 RPC 경로 또는 텔레포트 수신 시).
- **완료 기준:** 생존 모드에서 탈락하면 1초 내 관전대로 이동해 마우스룩으로 경기를 구경할 수 있고, 배너가 뜨며, 다음 라운드에 정상 복귀. 무한 낙하 로그 없음.

### 4-5 연결 끊김 UX (신규, 권장)
- **문제:** 호스트가 나가면 클라 화면이 아무 안내 없이 굳는다(HUD는 SetIdle로 비워질 뿐).

- **완료 기준:** 호스트 강제 종료 시 클라가 5초 내 타이틀로 복귀하고 재접속이 가능.

### 4-6 SfxManager + BGM
- **구현:** 신규 `Assets/Scripts/UI/SfxManager.cs`, GameHUD와 같은 자동 부트스트랩(`RuntimeInitializeOnLoadMethod`) 싱글턴. 2D `AudioSource` 2개(SFX용 PlayOneShot, BGM용 loop). 클립은 `Assets/Resources/Audio/`에서 `Resources.Load<AudioClip>(id)`로 지연 로드+캐시, 파일이 없으면 조용히 무시(에셋 준비 전에도 컴파일/실행 안전). 공개 API: `SfxManager.Play(string id, float vol = 1f)`, `SfxManager.PlayBgm(string id)`(0.5초 크로스페이드).
- **이벤트 훅 표:**

| id | 재생 시점 (호출 위치) |

- **에셋 소스:** Kenney "Interface Sounds"/"Impact Sounds"(CC0, 표기 불필요), BGM은 Kenney "Music Loops"(CC0) 또는 Kevin MacLeod(CC-BY, README에 크레딧 필수). 페이즈별 BGM은 로비/준비=느긋, 플레이=업템포, 결과=팡파레 3트랙.
- **완료 기준:** 위 표의 훅 중 필수 6개(click/vote/roulette_tick/roulette_win/splash/fanfare) 이상이 실제 재생되고, 클립 부재 시에도 예외 없음.

## 5. 서버 배포

**목표:** 데모 환경에서 실제로 여러 사람이 접속 가능하게 만든다. 먼저 **배포 방식을 결정**하는 것이 이 항목의 첫 작업이며, 나머지 폴리싱 작업의 테스트 방식에도 영향을 준다.

### 방식 비교

| 방식 | 작업량 | 장점 | 단점 |
|---|---|---|---|
| A. 현재 그대로 (LAN 직접 IP) | 없음 | 이미 동작함 | 발표장이 같은 네트워크가 아니면 접속 불가 |
| B. Unity Relay (Unity Gaming Services) | 적음 | NGO가 UTP+Relay 네이티브 지원, Steam 없이 NAT 우회, 무료 티어 | Unity Dashboard 프로젝트 연결 필요 |
| C. KCLOUD 헤드리스 전용 서버 | 많음 | 상시 접속 가능, README 구현명세서의 "전용 서버(DGS)" 방향과 일치 | KCLOUD VM은 기본 22/80/443 포트만 허용, 게임 UDP 트래픽 위해 보안그룹에서 커스텀 포트 개방 필요 |
| D. Facepunch/Steam 릴레이 | 큼 | 원래 설계 목표, 출시 대비 | Steamworks 인프라 필요, 데모엔 과함 |

**권장:** B(Unity Relay)를 데모용 기본으로, 시간 여유가 있으면 C를 병행. D는 출시 단계로 보류(README의 "필수" 오기는 정정 완료).

### 세부 작업 (B 기준)

- **5-1 UGS 프로젝트 연결:** Unity Dashboard에서 프로젝트 생성 → 에디터 `Project Settings > Services`에서 연결(무료 티어).
- **5-2 패키지 설치:** `com.unity.services.relay` + `com.unity.services.authentication`.
- **5-3 트랜스포트 교체:** `NetworkBootstrap.ApplyConnectionData()` 한 곳만 교체(설계상 이미 격리됨). 흐름: `UnityServices.InitializeAsync()` → `AuthenticationService.SignInAnonymouslyAsync()` → 호스트: `RelayService.CreateAllocationAsync(maxPlayers)` + `GetJoinCodeAsync()` → `UnityTransport.SetRelayServerData(...)`(dtls) → `StartHost()`. 클라: `JoinAllocationAsync(joinCode)` → 동일하게 SetRelayServerData → `StartClient()`. 접속 경로가 비동기가 되므로 버튼 핸들러를 async로.
- **5-4 Join Code UI:** 2-1 타이틀 화면에 통합. 호스트 화면에 발급 코드 6자리 표시(+클립보드 복사 버튼), 클라는 IP 입력칸 대신 코드 입력칸.
- **5-5 이종 네트워크 실측:** 팀원 각자 다른 네트워크(집/카페/핫스팟)에서 접속. **완료 기준: 서로 다른 공유기 뒤의 2인이 코드만으로 같은 매치 완주.**
- **5-6 (병행 시) KCLOUD:** Dedicated Server 플랫폼 Linux 헤드리스 빌드 + `-batchmode -nographics` 기동 + `StartServer()` 자동 실행 분기 + 보안그룹 UDP 7777 개방 + systemd 재시작 유닛. 선택.

## 6. 실행 순서 제안 (2인 팀 기준)

1. **(공동, 가장 먼저) 서버 배포 방식 결정**: LAN으로 충분한 데모라면 5절 작업 전체를 생략할 수 있으므로, 이걸 먼저 확정해야 이후 테스트 방식(로컬 2빌드 vs 실제 원격 접속)이 정해진다.
2. **버그/공정성 트랙 (신규, 반나절)**: 1-3 스폰 슬롯 수정 + 4-4 생존 탈락자 관전. 둘 다 멀티 플레이테스트의 전제조건이라 콘텐츠 작업보다 먼저.
3. **콘텐츠 트랙**: 맵 분위기 연출·밸런싱(1절) + 장애물 신규 타입(3절)
4. **필(feel) 트랙**: 캐릭터 모델·애니메이션·파티클·사운드(4절) + UI 폴리싱(2절)
5. **통합**: 새 맵 기준 전체 플레이테스트 → 밸런싱(장애물 개수/점수 비율 등) → Relay/서버 연동 최종 검증
6. **마무리**: 7절 빌드·데모 체크리스트 수행, README 회고(Keep/Problem/Try, 팀원 소감) 작성, README "개발 일정" 표를 실제 진행 상황에 맞게 갱신

시간이 부족해지면 README 구현명세서의 컷라인(왕관/코인 모드 → 3D 킬캠 → 실시간 리더보드 → 근접존 귀속)을 따르고, 이 문서 항목 중에서는 **선택 → 권장 순으로** 컷한다. 단 2번 트랙(버그/공정성)은 컷 불가.
