# 26s-w2-c1-07

## 공통과제 II : 협업형 실전 산출물 제작 (2인 1팀)

**목적:** 실시간 인터랙션, LLM Wrapper, Cross-Platform 중 하나의 옵션을 선택해 구현하며, 선택한 기술을 실제로 동작하는 형태의 산출물로 완성한다.

**선택 옵션:**

| 옵션 | 설명 |
|---|---|
| 실시간 인터랙션 | 사용자 간 상태 변화, 실시간 데이터 흐름, 스트리밍 응답 등 실시간성이 드러나는 기능을 구현 |
| LLM Wrapper | LLM API를 활용하여 AI 기능이 포함된 산출물을 구현 |
| Cross-Platform | 하나의 산출물을 여러 실행 환경에서 사용할 수 있도록 구현* |

> *데스크톱 앱 ↔ 모바일 앱; 혹은 다른 폼팩터에서의 앱; 웹만/웹 기반 프레임워크(Electron, Tauri 등) 대신 다른 프레임워크를 시도해보는 것을 적극 권장

**결과물:** 선택한 옵션이 적용된 작동 가능한 산출물, 실행 가능한 코드, 시연 자료 및 관련 문서

## 구현 명세서

이 문서는 7개 서브시스템 명세를 하나의 마스터 명세서로 통합하고, 2인·7일·TCP(WebSocket)·Cloudflare Tunnel 제약 하에서 실현 가능성을 재검수해 확정한 최종본이다. 우선순위는 세 단계로 표기한다. **필수**는 7일 MVP 시연을 성립시키는 코어(없으면 데모 자체가 불가), **권장**은 시간이 되면 완성도·안정성을 끌어올리는 항목, **선택**은 여유가 남을 때만 착수하는 확장이다. 서브시스템 간 중복 항목은 단일 소유자로 병합하고 나머지는 상호 참조로 연결했다. **필수 시연 목표 동시 접속은 20~24명**(안정 하한 12명, 스트레치 30~50명, 야심 목표 100명은 확장 계층). 각 필수 항목이 어느 Day에 대응되는지는 문서 하단 "일정 매핑"에 명시한다.

> 검수 결과 초안은 전반적으로 견고했다. 확정 과정에서 조정한 핵심 4가지: (1) **플레이어 접속·닉네임·로비 입장**을 명시적 필수 항목으로 추가(초안 누락), (2) 하트비트를 재접속 복원과 분리해 **필수로 승격**(Cloudflare Tunnel 유휴 종료가 시연을 직접 위협), (3) **라운드 중 실시간 Top-N 점수판을 권장으로 강등**(개인 순위 HUD만 필수 유지), (4) **룰렛/투표 풀을 구현 완료 모드로 한정**해 미구현 주제 투표를 방지. 상세는 하단 "검수 조정 요약" 참조.

### 1. 실시간 네트워킹 & 상태 동기화

| 구현 요소 | 설명 | 우선순위 |
|---|---|---|
| 넷코드 권위 모델 (클라 권위 이동 릴레이 + 서버 권위 게임 원장) | 이동은 각 클라가 로컬 물리로 시뮬하고 위치·yaw·상태플래그를 보내면 서버는 **이동을 재시뮬하지 않고 릴레이만** 한다(리컨실리에이션 통째 회피). 서버 틱은 이동 재계산이 아니라 게임 로직(페이즈·모드 onTick·사망 판정·스냅샷 조립)만 수행한다. 페이즈·점수·코인/왕관 소유권·골인·투표·룰렛 등 게임 로직은 서버가 단일 진실원으로 중재하고, 경쟁 자원은 '선착순 1인' 원자 처리. 이동 속도 상한·순간이동·맵밖 클램프만 경량 방어. **모든 후속 항목의 전제라 첫 결정으로 확정.** | 필수 |
| 방/세션 구조 + 서버 게임 루프 + 페이즈 상태머신 | MVP는 전역 단일 룸(전원 입장)이되 RoomManager는 다중 룸 확장 가능하게 추상화(인스턴스 1개, cap은 config). 룸 상태 = 플레이어 레지스트리·장애물 리스트·투표 집계·현재 주제·페이즈 FSM. 룸당 `setInterval` 게임 루프가 스냅샷 조립·브로드캐스트와 페이즈 타이머를 구동. 페이즈 전이 권위 및 시각 동기화는 §3 매치 상태 머신으로 단일화. | 필수 |
| 플레이어 접속 & 로비 입장 (닉네임·스폰·초기 월드 스냅샷) | 접속 시 클라가 닉네임을 보내면 서버가 길이·중복·금칙 검증 후 `playerId`+`sessionToken` 발급, 색/스폰 지점 배정, 레지스트리 등록. 응답으로 **현재 phase·endsAt + 전체 월드 스냅샷(플레이어·장애물·엔티티)**을 동봉해 진행 중 라운드에도 즉시 합류. 로비 화면 UI는 §6. **게임 성립의 입구라 초안에서 누락돼 있던 필수 코어.** | 필수 |
| 상태 브로드캐스트 & 스냅샷 배치 | 서버 틱 10~15Hz로 룸 상태를 '1 클라당 1 배치 메시지'로 묶어 전송(플레이어별 개별 메시지 금지 → 클라당 O(1)/틱). 고빈도 위치 채널과 저빈도 신뢰 이벤트 채널(투표·페이즈·배치=JSON)을 분리, 클라 송신 15Hz + lastSeq 에코 흐름 제어. MVP는 JSON, 30명+ 스트레치에서 바이너리·델타 전환(§7 최적화 항목과 동일). | 필수 |
| 클라이언트 보간 & 원격 플레이어 렌더링 | 원격 플레이어는 렌더 지연 버퍼(~100~120ms=약 2틱)로 스냅샷 사이를 lerp/slerp 보간해 저틱을 60FPS로 부드럽게 표현. 스냅샷 2~3개 링버퍼, 리스폰/텔레포트는 스냅 처리, 유실 구간은 짧게 외삽 후 정지. 로컬 플레이어는 클라 권위라 예측/리컨실리에이션 불필요. (렌더 구현체는 §6와 단일화.) | 필수 |
| 규모 산정 & 부하 테스트 봇 하니스 | 헤드리스 봇 클라이언트(ws 접속·랜덤워크 송신·스냅샷 수신)를 Day2에 함께 제작해 N=10/30/50의 부하·지연·손실을 반복 실측하며 틱레이트·압축을 튜닝한다. **2인 팀이 다인원 규모를 검증할 유일한 수단이라 필수.** | 필수 |
| 하트비트 · 킵얼라이브 | 앱레벨 하트비트(2~3s ping / 8~10s 타임아웃)로 죽은 소켓을 탐지하고, 동시에 **Cloudflare Tunnel 유휴 종료(~100s)를 회피**한다. 시연 도중 유휴 연결이 끊기면 데모가 즉시 깨지므로, 초안에서 재접속 복원과 묶여 권장이던 것을 **필수로 분리·승격.** | 필수 |
| 재접속 유예 복원 (토큰 기반) | 끊김 시 30~60s 유예 보존, 동일 sessionToken 재접속이면 점수·소유 장애물 복원 + 월드 스냅샷 재전송. 유예 초과 이탈자는 관전/유령 처리하되 배치 장애물·점수는 유지. 하트비트 없이도 시연은 성립하므로 복원 로직 자체는 권장선. | 권장 |
| 관심영역(AOI) 인터레스트 관리 | 맵을 균일 그리드 셀(20~30m)로 분할, 스냅샷 조립 시 자기 셀+인접 8셀 내 플레이어만 포함해 조립 비용을 O(N²)→근사 O(N·k)로. 셀 이동 시 enter/leave 이벤트로 원격 엔티티 생성/파괴, 전역 이벤트는 AOI 무시. **30명+ 스트레치에서만 투자**(코어 안정 후 착수). | 선택 |

### 2. 준비 페이즈 (장애물 배치 + 주제 투표 + 룰렛)

| 구현 요소 | 설명 | 우선순위 |
|---|---|---|
| 장애물 배치 명령 파이프라인 & 데이터 모델 | 클라는 `obstacle:place{type(WALL/CYLINDER/GHOST_WALL), pos, yaw}`만 요청하고 id 발급·검증·확정은 전적으로 서버가 담당. 서버 권위 상태 `obstacle{id, type, pos, yaw, ownerId, createdAt}`를 Map 보관, 통과 시 `obstacle:added` 브로드캐스트·실패 시 요청자에게 `obstacle:rejected{reason}`. yaw는 15°×12단계 양자화. 신규 접속·라운드 전환 시 `obstacle:snapshot`으로 전체 재동기화. | 필수 |
| 서버 권위 배치 제한 규칙 (개수·구역·충돌) | (개수) 1인당 3개 고정, ownerId 카운트로 초과 거부. (구역) 스폰 앞 5m·골 뒤 5m·강물 등을 AABB 금지 볼륨으로 판정. (충돌) 각 장애물 AABB로 겹침/0.5m 미만 간격 거부, pos는 서버에서 0.5m 그리드 스냅. 모든 판정은 서버 전용, 클라 프리뷰 색(빨강/초록)은 UX일 뿐 신뢰하지 않음. | 필수 |
| 투명 벽 특수 장애물 (공유 결정론 콜라이더 & 킬 귀속 검증) | 투명 벽은 별도 물리가 아니라 **일반 솔리드 콜라이더 + `visualStyle:ghost` 플래그**. 서버가 실제 충돌체 데이터를 그대로 브로드캐스트(모두 같은 벽 → 공정)하고 렌더만 클라가 투명/기만적으로 그린다. 점수용으로는 클라 `player:died{pos, cause, nearObstacleId}` 보고를 서버가 임계 반경 내 실제 존재 여부로 검증 후 소유자에게 귀속(허위 킬 차단). | 필수 |
| 주제 투표 집계 & 실시간 브로드캐스트 | 1인 1표, 페이즈 종료 전 변경 가능. `vote:cast{topicId}` → 서버가 playerId→topicId Map으로 권위 집계(재투표 덮어쓰기). 250~500ms 디바운스 `vote:update{counts[]}`로 실시간 판세를 보여주되 브로드캐스트 폭주 억제. 페이즈 종료 시 잠금 후 최종 counts를 룰렛 입력으로 확정. **투표 대상은 구현 완료된 모드만 노출**(MVP 초기 3모드, 코인/왕관 완성 시 추가). 미구현 주제 당첨을 원천 차단. | 필수 |
| 투표율 가중 룰렛 (서버 결정·직전 주제 제외·연출 동기화) | **서버가 결과를 먼저 정하고 클라는 연출만 재생.** 가중치 `w_i = votes_i + 1`(라플라스 스무딩), 직전 당첨 주제는 `w=0` 제외(장애물·투표 유지). 누적합 가중 랜덤으로 `winningTopicId` 확정 후 `roulette:result{winningTopicId, spinSeed, startAt, durationMs≈5000}` 브로드캐스트, 클라는 시계 오프셋에 맞춰 해당 칸에 정확히 정지. 룰렛 후보 = 구현 완료 모드 풀. 다음 라운드 권위 상태는 오직 서버 winningTopicId 사용. | 필수 |
| 준비/룰렛 페이즈 상태 머신 & 서버 타이머 동기화 | 서버 권위 FSM이 `phase:changed{phase, endsAt(서버시각)}`을 전이 시 1회 브로드캐스트, 클라는 endsAt+오프셋으로 카운트다운을 각자 계산(매초 전송 없음). PREP 종료 시 배치·투표 잠금 → ROULETTE 전이. 재접속 시 현재 phase+endsAt+스냅샷 동봉. **§3 매치 FSM과 동일 축**이라 그쪽으로 단일 구현(초기엔 setTimeout 스텁으로 병렬 개발 가능). | 권장 |
| 배치 비용제·이동형/트램펄린·라운드별 재배치 확장 | (비용) 인당 예산 10p + 코스트(벽1·원기둥1·투명벽3)로 개수제 대체해 투명벽 남발 억제. (이로운 배치물) `behavior` 필드(bounce/patrol) 확장. (재배치) 라운드 사이 짧은 재배치 창(추가/제거만). 모두 기존 데이터 모델의 필드/규칙 확장으로 흡수되게 설계. | 선택 |

### 3. 라운드 시스템 & 게임 모드

| 구현 요소 | 설명 | 우선순위 |
|---|---|---|
| GameMode 공통 인터페이스 & 모드 레지스트리 | 서버측 인터페이스 `GameMode{ config, onStart, onTick(ctx,dt), onPlayerDeath, onEnd → raw[] }`를 초반에 확정. `config{id, displayName, playDurationSec, respawnPolicy, winType, usesEntities}`. 상태 머신은 구체 모드를 전혀 모르고 훅만 호출 → **신규 모드 추가 = 파일 1개 + 레지스트리 등록.** '같은 맵 다른 게임'·'주제 추후 추가' 확장성의 뿌리. | 필수 |
| 매치 상태 머신 + 서버 틱 루프 + 라운드 타이머 | 서버 권위 FSM으로 6페이즈(PREP→ROULETTE→PLAY→HIGHLIGHT→loop→RESULT) × 3라운드 관리. 진입 시 `phaseChanged{phase, round, endTimestamp}` 1회 브로드캐스트 → 인원과 무관하게 타이머 트래픽 0(주기적 clock-sync로 드리프트 보정). PLAY는 15~20Hz 고정 틱이 현재 모드 onTick+사망 판정, 스냅샷은 10~15Hz. 타이머 만료/조기종료 시 onEnd→점수→전이, 다음 modeId를 레지스트리에서 주입(직전 주제 제외는 §2 룰렛이 처리). **전체 플로우의 실행 컨텍스트.** | 필수 |
| 사망/리스폰/관전 시스템 | 서버가 매 틱 Y를 `killPlaneY`와 비교하고 해저드 볼륨 충돌로 사망 권위 판정 → `playerDied{playerId, pos, cause, killerObstacleId?, t}` 이벤트 버스 발행(점수·하이라이트·장애물영향이 구독). 리스폰은 config 종속: respawn 모드는 2~3s 후 체크포인트 부활+1.5s 무적, spectate 모드는 관전 전환. 사망지점 근처 장애물로 killerObstacleId 채움. **3개 모드와 하이라이트 지표의 공통 데이터 소스.** | 필수 |
| 핵심 3모드 판정 로직 (레이스 / 최고높이 / 생존) | 신규 엔티티 0개로 세 승리 아키타입 커버. (1) 목표지점: 골 트리거 진입시각, raw=완주자 `(BIG−finishMs)`·미완주자 경로 진행거리. (2) 최고높이: onTick으로 peakY 갱신, raw=종료시점 Y(동점은 peakY 도달시각). (3) 생존: playerDied 구독으로 survivalMs 확정. 위치·사망 데이터만 소비 → Day5 하루에 3개 완성. (레이스 경로거리는 맵 체크포인트 순번 데이터 필요.) **최소 1개(레이스)는 절대 시연 경로.** | 필수 |
| 코인 수집 모드 (신규 엔티티) | `coin{id, pos, taken}` 배열 스폰. 서버 권위 픽업: 반경 진입 시 `taken=true` 후 `coinTaken{coinId, playerId}` 브로드캐스트, 클라는 즉시 반응 후 서버값으로 정정. onEnd raw=coinCount. 스냅샷에 코인 상태 델타만 실음. 옵션: N초 후 재생성으로 후반 역전 유도. (준비페이즈 장애물과 겹치지 않게 맵 데이터 조율.) | 권장 |
| 왕관 오래 들기 모드 (다중 배치) | `crown{id, pos, holderId|null}` K개 스폰. 픽업 시 holder 지정·브로드캐스트, onTick에서 홀더 있는 모든 왕관마다 dt를 `crownHoldMs`에 가산(다중 소지 시 배수 누적). 사망 시 사망지점 드롭, 방치 시 X초 후 스폰지점 복귀. onEnd raw=crownHoldMs. **MVP는 K=1로 검증 후 여유 시 K=3 확장(다중배치는 선택).** 상태 수가 가장 많아 이 서브시스템 최고 리스크. | 권장 |
| 라운드 점수 집계 & 3라운드 누적 순위 | 각 라운드 onEnd raw 배열을 순위→포인트로 환산해 누적 테이블에 가산, RESULT에서 최종 순위 브로드캐스트. **정규화 산식·장애물 어트리뷰션·70:30 비율의 상세는 §4로 단일화**(여기서는 raw 공급과 파이프라인 연동만 담당). raw만 내면 자동 편입되므로 신규 모드 확장성 확보. | 필수 |

### 4. 점수 시스템

| 구현 요소 | 설명 | 우선순위 |
|---|---|---|
| 모드별 클리어 점수 정규화 산식 (라운드당 0~1000 공통 스케일) | 서버가 라운드 종료 시 authoritative하게 0~1000 정규화. (1)레이스: 완주 `500+500*(M−r)/max(M−1,1)`, 미완주 `500*진행률`. (2)최고높이: `1000*clamp(h_peak/H_max,0,1)`(peak 사용으로 낙사 왜곡 방지). (3)생존: `1000*t_survive/T`. (4)코인: `1000*c/c_max`(상대 정규화, c_max=0이면 전원 0). (5)왕관: `1000*hold_t/T`. 라이브 지표는 게임루프가 상시 집계, 클라는 표시만. **이질적 5모드의 3라운드 누적 공정성의 심장.** | 필수 |
| 장애물 소유자 태깅 & last-touch 사망 어트리뷰션 | 배치 시 ownerId 부여, 충돌 감지 시 `lastContact={obstacleId, ownerId, t}` 갱신. 사망 시 `now − lastContact.t ≤ 1.5s`면 소유자에게 kill +1(last-touch), 접촉했으나 미사망이면 disturb +1. self-kill은 제외. **비접촉 간접 인과는 의도적 배제**(오탐·비용). 기존 충돌 판정에 타임스탬프 1개 추가면 되어 안정 구현 가능. | 필수 |
| 장애물 영향 점수 환산 & 라운드 총점 합성 (클리어 70 : 장애물 30) | `raw_impact = kill*60 + disturb*12`, `obstacle_score = min(300, raw_impact)`(절대 상한 → 2차 정규화 불필요). 클리어 기여 = `clear_1000*0.7`(0~700). **라운드 점수 = round(clear_1000*0.7) + obstacle_score**(0~1000). kill 60·disturb 12·상한 300은 5킬 만점 밸런스 초안. 실력 부족자도 배치로 최대 30% 만회 → 접근성. | 필수 |
| 3라운드 누적 점수 & 최종 순위 집계 | 서버가 `scoreState{round[1..3]={clear,obstacle,total}, cumulative}`를 authoritative 유지. 라운드 확정 시 total을 cumulative 가산 후 breakdown 포함 결과창을 1회 브로드캐스트(저빈도라 100명도 부담 없음). 최종 = cumulative 내림차순, 타이브레이커 (1)라운드 1위 횟수 → (2)누적 kill → (3)입장 순. **게임 최종 산출물, 재접속·이탈에도 점수 보존.** | 필수 |
| 개인 순위 HUD (myRank·myScore·myLiveMetric) | 라운드 중 각 클라에 자기 순위·점수·모드별 라이브 지표만 저빈도 개인화 전송 → 페이로드가 인원 무관 상수 크기. 자기 위치를 아는 최소 게임 피드백이라 필수(표시는 §6 HUD). | 필수 |
| 실시간 Top-N 점수판 (전체 리더보드 개인화 스냅샷) | 위 HUD를 확장해 `{top10, myRank, ...}`를 2~4Hz 개인화 전송(O(N²) 방지). 파티게임 긴장감을 높이지만 최종 결과창과 개인 HUD로 데모는 성립하므로 **초안의 필수에서 권장으로 강등.** 순위 급변 허용이라 보간 생략. | 권장 |
| 근접존(비접촉) 낙사 어트리뷰션 확장 & 방해 세분화 | 낙사 반경 R(≈3m) 내 소유자 장애물 존재 + K초 근방 체류 + 이동벡터가 벽 방향이면 부분 크레딧(kill*0.5). 공간 질의·방향 휴리스틱 튜닝 필요. disturb를 넉백/감속/경로이탈로 세분화. **오탐 위험이 커 여유 시에만**(투명벽도 대개 벽면 접촉이 먼저라 §4-2로 상당수 포착). | 선택 |

### 구현 규모 및 현실성

MVP 시연의 **필수 목표 동시 접속은 20명 안팎**(안정 검증 하한 12명, 실용 목표 20~24명), **스트레치 30~50명**, **야심 목표 100명**의 3단계로 분리한다. 근거는 TCP WebSocket 브로드캐스트가 팬아웃 O(N)·총대역 O(N²)이고 단일 Node 프로세스+Cloudflare Tunnel에서 100명은 AOI 컬링·바이너리·델타 압축이 선행돼야 하나 2인·7일로는 구현·QA 불가라는 점, 그리고 20~30명이면 '투명벽에 수십 명 우수수 낙사' 개그와 숏폼 임팩트가 충분히 성립한다는 점이다. 야심 100명과의 격차는 **아키텍처에 확장 여지만 남기고 필수 범위에서 제외**하는 방식으로 다룬다: 룸 cap을 config로 빼고, 스냅샷 배치·개인화 채널·룸 추상화를 처음부터 넣어 두되, 바이너리 인코딩·AOI·델타 압축은 선택 스트레치 계층으로 미룬다. 규모 주장은 반드시 헤드리스 봇 하니스(Day2)로 N=10/30/50을 실측해 검증한다.

**최소 시연 경로 (골든 패스).** 다른 모든 것이 무너져도 이 한 줄은 끝까지 동작해야 한다: `접속·닉네임 → 준비(장애물 배치+투표) → 룰렛 → 레이스 1모드 플레이 → 라운드 점수 → 룰렛 재추첨 → 2·3라운드 → 최종 누적 순위 + 2D 하이라이트 카드`.

**컷 순서 (Day5~6 슬립 시).** 아래 순서로 먼저 버린다: ① 왕관·코인 모드(레이스 1모드로 데모 성립) → ② 3D 킬캠(2D 스탯카드 유지) → ③ 실시간 Top-N 점수판(개인 순위 HUD 유지) → ④ 근접존 어트리뷰션. **보호선**: 완주 가능한 1개 모드 + 준비/룰렛 + 3라운드 누적 순위 + 2D 하이라이트 카드는 어떤 경우에도 사수한다.

| 실시간 통신 | Socket.IO (WebSocket 전용, polling off) | Cloudflare Tunnel이 UDP/WebRTC/QUIC 차단 → TCP WebSocket 강제, room·재접속·하트비트 기본 제공 |
| 넷코드 방식 | 클라 권위 이동 릴레이 + 서버 권위 게임 원장, 배치 스냅샷 | 서버 물리 재시뮬·리컨실리에이션 회피가 7일 일정의 결정적 절충 |
| 틱레이트 | 서버 게임로직 틱 15~30Hz / 스냅샷 브로드캐스트 10~15Hz / 클라 송신 15Hz / 보간 버퍼 ~100~120ms | 저틱+보간으로 TCP 지터·헤드오브라인 흡수, 타이머는 endTimestamp 1회 전송 (서버는 이동을 재시뮬하지 않음) |
| 클라 렌더 | Three.js (raw) + Vite, InstancedMesh, DOM UI 오버레이 | 저폴리 컨셉·다수 인스턴싱·번들 최소, react-three-fiber 오버헤드 배제 |
| 물리 | 커스텀 키네매틱 컨트롤러 + 하이트맵 힙트필드 충돌 | 강체 시뮬 불필요, 지면 충돌 O(1), 클라·서버 동일 콜라이더 공유 |

| 프로젝트 구조 | pnpm workspace 모노레포 (shared / server / client) | shared 프로토콜 타입을 단일 진실원으로 두어 계약을 컴파일러가 강제 |

### 검수 조정 요약 (초안 대비 변경점)

- **[추가·필수] 플레이어 접속 & 로비 입장** (§1): 닉네임 검증·playerId/sessionToken 발급·스폰·초기 월드 스냅샷. 초안에 게임의 '입구'가 명시적 필수로 없어 추가.
- **[승격·필수] 하트비트·킵얼라이브** (§1): 재접속 복원과 분리. Cloudflare Tunnel 유휴 종료(~100s)가 시연 중 연결을 끊어 데모를 직접 위협하므로 필수. 유예 복원 로직은 권장 유지.
- **[강등·권장] 실시간 Top-N 점수판** (§4): 최종 결과창 + 개인 순위 HUD로 데모가 성립하므로 전체 리더보드 실시간 푸시는 권장으로 하향. 개인 순위 HUD는 필수 유지.
- **[명확화] 룰렛/투표 주제 풀** (§2): 구현 완료 모드만 노출해 미구현 주제 당첨을 차단. MVP는 3모드로 3라운드 순회.
- **[정합성] 서버 틱 정의**: '서버 물리/시뮬 틱'이 이동 재시뮬로 오독되지 않도록, 서버는 이동을 재계산하지 않고 게임 로직(모드 onTick·사망 판정·스냅샷 조립)만 돌린다는 점을 §1·§7에 명시. 100명 서버 물리 주장은 초안에도 없었고 20~24명 릴레이 모델로 일관.
- **[추가] 일정 매핑·골든 패스·컷 순서**: 필수 항목을 Day2~7에 고정하고, 최소 시연 경로와 슬립 시 컷 순서를 명문화해 일정 정합성과 리스크 컷라인을 확보.

#### KCLOUD VM / Cloudflare Tunnel 환경별 주의사항

| 환경 | 사용 가능(권장) 기술 | 포트/조건 | 주의할 기술 |
|---|---|---|---|
| **로컬 / 일반 VM** | HTTP/REST, WebSocket, Socket.IO, SSE, TCP Socket, gRPC Streaming, WebRTC, QUIC/WebTransport 등 대부분 가능 | 직접 포트 개방 가능. 예: 3000, 5000, 8000, 8080, 9000 등. 외부 공개 시 방화벽/보안그룹/공인 IP 설정 필요 | WebRTC는 STUN/TURN 필요 가능. QUIC/WebTransport는 HTTP/3 · UDP 지원 필요 |
| **KCLOUD VM (VPN 내부)** | HTTP/REST, WebSocket, Socket.IO, SSE, WebRTC 시그널링 | 접속 기기 VPN 필요. 기본 허용 포트: **22, 80, 443**. 개발 포트(3000, 8000, 8080 등)는 직접 접근 제한 가능 | TCP Socket은 포트 제한 있음. gRPC는 HTTP/2 설정 필요. WebRTC 미디어·UDP·QUIC/WebTransport 비권장 |
| **KCLOUD VM + Tunnel** | HTTP/REST, WebSocket, Socket.IO, SSE, WebRTC 시그널링 | VM의 `localhost:<port>`를 도메인에 연결. `localPort`는 **1024~65535**. 예: 3000, 8000, 8080 가능 | 순수 TCP Socket, UDP, WebRTC 미디어/DataChannel, QUIC/WebTransport 불가. gRPC 보장 어려움 |
| **외부 서비스 + 우리 도메인** | HTTP/REST, WebSocket, Socket.IO, SSE, WebRTC 시그널링 | Vercel/Netlify/Railway/Render/AWS/GCP 등에 배포 후 CNAME/A 레코드 연결. 보통 외부는 **443** 사용 | WebSocket/gRPC/TCP/UDP는 플랫폼 지원 여부 확인 필요. 서버리스 플랫폼은 장시간 연결 제한 가능 |
| **서버 없이 외부 SaaS 사용** | Supabase Realtime, Firebase, Pusher/Ably, LLM API Streaming | 직접 포트 관리 불필요. 각 서비스 SDK/API 사용 | 커스텀 TCP/UDP 서버 구현 불가. WebRTC는 STUN/TURN 필요할 수 있음 |
